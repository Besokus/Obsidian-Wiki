# MySQL 慢查询

## 问题描述
SQL 执行时间超过预期，可能导致接口超时、连接池耗尽、数据库负载升高。MySQL 慢查询日志可以记录超过 `long_query_time` 的 SQL。

## 为什么会发生
```text
1. 缺少合适索引。
2. 索引失效或回表过多。
3. 返回或扫描数据量过大。
4. 排序、分组、分页成本高。
5. 锁等待或事务持有时间过长。
6. SQL 写法导致优化器选择低效执行计划。
```

## 影响
- 可用性：慢 SQL 堆积可能拖垮数据库。
- 性能：CPU、磁盘 IO、内存压力上升。
- 用户体验：接口响应时间变长。
- 稳定性：连接池、线程池可能被阻塞。

## 典型场景
- 列表页深分页。
- 大表无索引条件查询。
- `select *` 导致回表和网络传输变多。
- 高并发更新同一批记录导致锁等待。

## 定位路径
```text
1. 开启或查看慢查询日志。
2. 找到具体 SQL、耗时、扫描行数。
3. 用 EXPLAIN 分析执行计划。
4. 判断是索引、SQL 写法、锁等待还是数据量问题。
5. 做最小改动并复测。
```

## 常用命令
```sql
show variables like 'slow_query_log';
```

配置示例：
```ini
slow_query_log=1
long_query_time=2
```

## 解决方案
| 方案 | 适用场景 | 代价 / 风险 |
|---|---|---|
| 补充或调整索引 | where/order/group 条件稳定 | 写入变慢，索引占空间 |
| 改写 SQL | 不必要回表、函数运算、深分页 | 需要验证结果一致 |
| 限制返回列 | `select *` 导致读取过多 | 需要改调用方字段依赖 |
| 拆分大查询 | 单次处理数据量过大 | 业务逻辑更复杂 |
| 排查锁等待 | 更新慢或事务堆积 | 需要结合事务和锁分析 |

## 我的理解
慢查询不是一个单点问题，而是查询计划、索引设计、数据规模和并发状态共同作用的结果。定位时先拿证据，再改 SQL 或索引。
