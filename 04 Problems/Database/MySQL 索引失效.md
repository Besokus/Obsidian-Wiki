# MySQL 索引失效

## 问题描述
明明创建了索引，但查询执行时没有使用索引，或者只使用了联合索引的一部分，导致扫描行数增多、查询变慢。

## 为什么会发生
```text
1. 在索引列上做函数、表达式或运算。
2. 字符串字段查询时没有加引号，发生隐式类型转换。
3. like 使用头部模糊匹配，例如 '%abc'。
4. or 条件中有列没有索引。
5. 联合索引不满足最左前缀法则。
6. 优化器判断走索引比全表扫描更慢。
```

## 影响
- 正确性：通常不影响结果正确性。
- 可用性：慢查询堆积可能拖垮数据库。
- 性能：扫描行数增加，CPU 和 IO 上升。
- 用户体验：接口延迟变高。

## 典型场景
- 搜索接口使用 `like '%keyword%'`。
- 手机号、状态等字段类型和查询参数类型不一致。
- 复杂筛选接口随意组合 `or` 条件。
- 联合索引设计和实际查询条件顺序不匹配。

## 定位路径
```text
1. 用慢查询日志或监控找到慢 SQL。
2. 用 EXPLAIN 查看 type、key、rows、filtered、Extra。
3. 对照索引定义和 where/order by/group by 条件。

```

## 解决方案
| 方案 | 适用场景 | 代价 / 风险 |
|---|---|---|
| 改写 SQL | 函数、隐式转换、模糊匹配导致失效 | 需要确认业务语义不变 |
| 调整联合索引 | 查询条件稳定且高频 | 增加写入维护成本 |
| 拆分 or 查询 | or 中部分列无索引 | SQL 复杂度上升 |
| 使用覆盖索引 | 高频查询只返回少量字段 | 索引可能变宽 |

## 最小可行修复
先用 [[EXPLAIN]] 确认索引失效原因，再做最小 SQL 改写或补充索引，不要直接盲目加索引。

## 长期治理
- 为高频查询保留 EXPLAIN 记录。
- 建索引前确认查询模式，而不是按字段名猜。
- 定期检查慢查询日志。

## 我的理解
索引失效本质上是“查询条件无法匹配索引的有序结构”或“优化器认为索引不划算”。排查时要看执行计划，不要只看表上有没有索引。
