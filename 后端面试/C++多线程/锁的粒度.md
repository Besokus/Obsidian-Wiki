# 🌟锁的粒度

**锁粒度指的是加锁操作所覆盖的代码 / 数据范围的大小**。

**粗粒度锁**：对整个共享资源 / 整个方法加锁，覆盖范围大，并发能力差，但实现简单，无需考虑多锁协调问题；

**细粒度锁**：对共享资源的不同部分分别加锁，覆盖范围小，并发能力强，但实现复杂度高，还可能引入新问题（如死锁）。



**ConcurrentHashMap 的细粒度锁**

Java 中的`HashMap`是非线程安全的，`Hashtable`是线程安全的，但它用的是**粗粒度锁**（对整个哈希表加锁）—— 任何线程操作 Hashtable 时，其他线程都要等待，并发性能差。

而`ConcurrentHashMap`（JDK1.8）采用了**分段锁（细粒度锁）**设计：

1. 将哈希表拆分成多个 “桶段（Segment）”，**每个桶段有独立的锁**；
2. 线程操作某个桶段的数据时，只锁住该桶段，其他桶段仍可被其他线程操作；
3. JDK1.8 进一步优化为 “CAS + 同步块”，对单个链表 / 红黑树节点加锁，粒度更细。
