# 为什么条件变量必须和 mutex 一起使用？

**条件变量与 mutex 绑定，是为了避免 “竞态条件”和"虚假唤醒" ，保证条件检查和线程等待的原子性，最终确保线程间通信的安全与可靠**。



条件变量（Condition Variable, CV）本身并不包含条件，它只是一个“通知机制”。真正的条件（比如 `queue.empty()` 或 `count > 0`）通常被称为**谓词（Predicate）**。

- **竞态风险**：这个谓词通常是一个被多个线程共享的变量。
- **必须加锁**：为了防止多个线程同时读写这个共享变量，必须使用 **Mutex（互斥锁）** 来保证对谓词检查和修改的原子性。



**防止“信号丢失”问题**

1. **线程 A（消费者）**：检查条件 `if (count == 0)`，发现成立，准备调用 `wait()`。
2. **线程 B（生产者）**：就在线程 A 还没来得及进入 `wait()` 的一瞬间，线程 B 修改了条件 `count = 1` 并执行了 `notify()`。
3. **结果**：由于此时线程 A 还没进入等待状态，这个 `notify()` 信号就丢失了。
4. **死锁**：随后线程 A 进入 `wait()` 并无限期休眠，即便此时条件已经满足，它也错过了那个唯一的通知信号。

**Mutex 的作用**：它确保了“检查条件”和“进入等待状态”这两个动作之间是受保护的，生产者的信号无法在这个间隙“插队”。



**应对“虚假唤醒”**

在多核处理器上，由于底层操作系统的实现，线程有时会在没有任何人 `notify` 的情况下被意外唤醒。

必须在 `while` 循环中检查条件，而不是 `if`

因为重新唤醒后需要**再次检查谓词**，所以必须**重新持有 Mutex** 才能保证检查过程的线程安全。
