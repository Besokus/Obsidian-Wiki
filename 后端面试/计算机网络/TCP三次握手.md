# 🌟TCP三次握手

**TCP 三次握手过程：**
 1️⃣ **第一次握手**：客户端向服务端发送 SYN 报文，表示希望**建立连接**，同时发送自己的初始序列号 seq = x。（**服务端确定客户端能发送信息**）
 2️⃣ **第二次握手**：服务端收到后，回复 SYN + ACK 报文，表示**同意建立连接**，同时确认客户端的序列号（ack = x+1），并发送自己的序列号 seq = y（**客户端确认服务端能收发消息**）
 3️⃣ **第三次握手**：客户端 → 服务端，发送 `ACK=1，ack=y+1`，表示确认连接成功。（**服务端确认客户端能收到消息**）

三次握手完成后，**双方**都确认：

- 自己的发送/接收能力正常；
- 对方的发送/接收能力正常。



**为什么是三次，不是两次？**

**核心回答逻辑：**

> 因为三次才能确保**双方的收发能力都被确认**。

📘 举例说明：

- 两次握手只能保证服务端能收到客户端的消息，但**无法确认客户端能否收到服务端的响应**；
- 如果只两次，服务端可能一直认为连接已建立，而客户端却没收到确认（服务端不知道客户端能否收到消息），导致**“半连接”**问题，**占用资源**。



**为什么不是四次？**

三次是**保证可靠性和效率的最小次数**。四次握手会**浪费一次网络往返**，没有必要。



**三次握手可以携带数据吗？**

> 第三次握手**可以携带数据**，前两次**不建议携带**。

- 第一、二次握手还未确认连接，携带数据会浪费资源（**可能被丢弃**）；
- 第三次握手客户端已收到服务端确认（客户端知道服务端能接收到），可以安全携带首个应用层数据。

⭐字节爱问点：**为什么不建议前两次携带数据？**
 → 因为在 TCP 实现中，未建立连接的包一般不会交给上层应用处理，会被丢弃。
