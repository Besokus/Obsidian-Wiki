# Cookie和Session是什么，token和他们相比有什么区别

Cookie 存在**客户端**，由浏览器自动携带，每次 HTTP 请求都会自动携带给服务器，常用于**保存用户标识或偏好**，不太安全。（服务器发送到浏览器、并由浏览器保存的一小段数据，大小有限，**无状态链接**）
Session 存在**服务器端**，浏览器通过携带 SessionID（通常存在 Cookie 中）来识别用户，相对安全。（保存在**服务器端的用户会话信息，有状态连接**）



**Token**

**携带用户身份和权限信息**，是服务器签发的“访问凭证”（常见为 JWT）

Token是**无状态连接**，服务器不存 Session，任意一台服务器都能校验 Token，天然支持负载均衡，适合用于分布式、微服务、前后端分离场景



**Token 在实际系统中是怎么“使用”的？**

1. 登录阶段（签发 Token）:用户提交用户名和密码，服务端验证通过，服务端生成 Token（写入用户信息 + 过期时间），返回给客户端

2. 客户端保存 Token：HTTP Header（推荐），Cookie

   ```http
   Authorization: Bearer <token>
   ```

3. 请求阶段（携带 Token）

4. **服务端校验 Token（关键流程）**：解析 Token，校验签名是否合法，校验是否过期从 Payload 中取出 userId / role，**识别当前用户身份并做权限控制**



**为什么 Token 能做到“无状态”**

因为用户身份信息已经包含在 Token 中，服务端只需验证签名即可识别用户，无需保存或查询会话状态。



**Token 是否可以作为网站登录时的“记住我”功能？**

可以。通过设置 Token 的较长有效期，或者使用“短期访问 Token + 长期刷新 Token”的方式，可以实现“记住我”功能。

- Access Token：短有效期（如 15 分钟）
- Refresh Token：长有效期（如 7 天 / 30 天）
- 用户勾选“记住我”时，刷新 Token 存得更久
