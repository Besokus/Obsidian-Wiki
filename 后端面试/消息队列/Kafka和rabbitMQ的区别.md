# Kafka和rabbitMQ的区别

| 特性         | RabbitMQ                                  | Kafka                                          |
| ------------ | ----------------------------------------- | ---------------------------------------------- |
| 类型         | 传统消息队列（Broker + Queue）            | 分布式日志系统（Distributed Commit Log）       |
| 设计目标     | 可靠传输、灵活路由                        | 高吞吐、持久化、顺序日志、流式处理             |
| 消息消费模式 | 消费者拉取/推送，消息消费后被删除（默认） | 消费者可重复读取，消息保留时间可配置           |
| 消息顺序     | 单队列内顺序                              | 分区内顺序，分区间无全局顺序保证               |
| 消息丢失     | 默认可靠（ACK机制）                       | 高吞吐下可保证“至少一次”或“精确一次”，需要配置 |



RabbitMQ 更偏向于**传统消息队列模型**，强调消息的可靠投递、灵活路由和低延迟处理，适合业务解耦、任务队列以及对实时性要求较高的场景，比如订单处理、异步任务和消息通知。

而 Kafka 更偏向于**分布式日志系统**，强调高吞吐、可扩展性以及消息的持久化和重放能力，适合大数据处理、日志收集、流式计算等场景。

从核心区别来看，RabbitMQ 是“**消息驱动**”的，消息被消费后一般就不再保留；而 Kafka 是“**日志驱动**”的，消息会长期存储，可以被多个消费者重复消费。

在实际选型中，如果更关注低延迟、复杂路由和业务解耦，一般会选择 RabbitMQ；如果更关注吞吐量、数据流处理和可扩展性，则更适合使用 Kafka。
