# 🌟讲一讲epoll模型,为什么它比 select / poll 更高效

**select / poll 的问题**

select 和 poll 本质是**轮询机制**，每次调用都需要把所有文件描述符从**用户态拷贝到内核态**，并遍历所有 fd 检查是否就绪。

当连接数很多时，这种 O(n) 的遍历和频繁的内核态/用户态拷贝**开销会非常大，效率较低**。



epoll 是 Linux 下的高性能 IO 复用机制，用于同时监控多个 socket 的事件。它通过**事件驱动机制**（而非select、poll的轮询机制），其内部采用**红黑树**管理监听 fd（文件描述符），并通过**就绪链表**管理已经就绪的fd，减少**无效检查**，相比 select 和 poll 在高并发场景下具有显著性能优势。





**epoll 为什么更高效**

相比 select/poll，epoll 的优势主要体现在：

1. **避免重复拷贝**

epoll 在注册 fd 时只拷贝一次，之后不需要每次调用都传入全部 fd

2. **事件驱动（而不是轮询）**

只返回就绪的 fd，时间复杂度接近 O(1)

3. **更好的扩展性**

使用红黑树管理 fd，没有 select 的 fd 数量限制

4. **就绪链表**

用户只处理“真正有事件”的连接



**epoll 支持水平触发和边缘触发两种模式**



**水平触发（LT）**：只要文件描述符（fd）处于 “**就绪状态**”（比如可读 / 可写），epoll_wait 就会**持续返回该事件，直到数据被处理完**；

只要就绪队列中的没有拿完就一直提醒，直到拿完为止



**边缘触发（ET）**：仅当 fd 的状态从**“未就绪”变为“ 就绪 ”时**，epoll_wait 才返回一次通知，后续即使有未处理的就绪数据，也不会再触发

只有当就绪队列状态状态改变（有事件加入等）时，才会发一次提醒



**ET 为什么更高效**

因为 ET 模式**减少了重复通知**，避免了 LT 模式下的 “频繁唤醒”，在高并发场景下可以**降低系统调用开销**。



**ET 模式为什么必须设置 fd 为非阻塞？**

如果 fd 是阻塞模式，当循环读 / 写时，若数据已读完，read/write 会阻塞在 fd 上，导致整个事件循环卡住，无法处理其他 fd 的事件；非阻塞模式下，数据读完会返回 **EAGAIN** 错误，可正常退出循环。



**ET 使用注意（重点）**

使用 ET 模式时必须：

1. **将 fd 设置为非阻塞**

2. **在一次事件处理中循环读/写，直到返回 EAGAIN**

否则容易导致数据没有读完，但又收不到下一次通知，从而造成“假死”。



**性能差异**

| 对比           | LT   | ET             |
| -------------- | ---- | -------------- |
| 通知次数       | 多   | 少             |
| 编程难度       | 低   | 高             |
| 性能           | 稍低 | 更高           |
| 是否容易丢数据 | 不会 | 会（读不干净） |



 **实际项目中怎么选？**

- 优先选 LT：开发成本低，不易出 bug，大部分中小并发场景足够用；
- 选 ET：高并发场景（如百万连接的网关、直播服务器），需极致性能，且能接受更高的编程复杂度（比如处理 EAGAIN、避免数据丢失）；
- 注意：ET 模式下要确保「一次性读完 / 写完所有数据」，否则未处理的数据会一直留在缓冲区，直到下次 fd 状态变化才会触发，可能导致数据延迟或丢失。
