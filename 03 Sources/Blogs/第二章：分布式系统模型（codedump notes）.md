# 第二章：分布式系统模型（codedump notes）

## 来源
[第二章：分布式系统模型](https://www.codedump.info/dist-system-cn/systemmodel/)

## 可信度判断
这是学习型博客章节，适合作为“系统模型的框架与术语”入口。涉及严格不可能性结论与同步性假设边界时，建议对照论文/教材进一步核验（例如 FLP、partial synchrony）。

## 它想解决什么问题
在设计算法/系统前，先明确运行环境的假设：网络如何通信、会坏到什么程度、延迟是否有上界。很多分布式问题不是实现细节，而是模型假设没说清导致的。

## 核心观点
```text
1. 设计分布式系统前必须先选定系统模型，否则讨论一致性/共识没有前提。
2. 通信模型关注链路可靠性抽象；故障模型是层级结构；时间模型决定你对“超时”的把握。
3. 工业界常见假设更接近“部分同步 + 崩溃/遗漏故障”，而不是拜占庭。
```

## 内容结构（按章节）
- 思想实验动机：
  - 两将军问题：展示不可靠通信下的“无限确认”陷阱。
  - 拜占庭将军问题：展示节点不可信/作恶带来的挑战（BFT 语境）。
- 系统模型三件套：
  - 通信模型：链路抽象（公平损失/可靠/认证）。
  - 故障模型：从 crash-stop 到 byzantine 的层级结构。
  - 时间模型：同步/异步/部分同步，决定超时是否能推断失败。

## 对开发者有什么用
- 写调用链治理策略时：先假设“网络不可靠 + 结果未知”，再谈 [[超时]] / [[重试]] / [[熔断]] / [[降级]]。
- 学共识算法时：先搞清楚它依赖的时间模型与故障模型，否则会把工程假设误认为算法性质。
- 做事故复盘时：用模型术语描述故障（遗漏/时序/响应/崩溃），比“它挂了”更接近根因。
