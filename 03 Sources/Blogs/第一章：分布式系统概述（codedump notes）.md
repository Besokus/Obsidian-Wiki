# 第一章：分布式系统概述（codedump notes）

## 来源
[第一章：分布式系统概述](https://www.codedump.info/dist-system-cn/introduction/)

## 可信度判断
这是一篇中文学习型博客，适合作为入门导读和问题框架来源。涉及分布式时间、CAP、一致性模型等较严格概念时，应继续对照论文、课程讲义或权威教材核验。

## 分布式计算的谬误
1. 网络是可靠的（The network is reliable）。
2. 延迟为零（Latency is zero）。
3. 带宽是无限的（Bandwidth is infinite）。
4. 网络是安全的（The network is secure）。
5. 拓扑结构不会发生改变（Topology doesn’t change）。
6. 存在一名管理员（There is one administrator）。
7. 传输成本为零（Transport cost is zero）。
8. 网络是同构的（The network is homogeneous）。

## 它想解决什么问题
这篇文章不是在教某个具体算法，而是在回答一个更基础的问题：为什么单机系统的直觉到了分布式系统里会失效，以及工程师应该先建立什么心智模型。

## 核心观点
```text
1. 分布式系统的本质是多个独立节点通过网络传递消息并协同工作。
2. 分布式系统的难点不在“多台机器”本身，而在网络不可靠、没有全局时钟、部分失效和副本一致性。
3. 从单机转向分布式，关键不是先学 Raft，而是先完成思维模型的转变。
```

## 什么是分布式系统
文章把分布式系统和单机系统做了直接对比。单机系统的优势是：

- 共享内存，本地调用延迟低。
- 有统一时钟，事件顺序容易判断。
- 一致性天然强，逻辑更简单。

但它的问题也很明显：

- 单点故障。
- 性能和容量都受单机上限限制。
- 只能依赖垂直扩展。

分布式系统则通过多个节点协作，换取：

- 横向扩展能力。
- 更高可用性和容错性。
- 更大的容量。
- 更贴近用户地域部署的能力。
- 对某些天然分布式业务的支撑能力。

## 文章强调的四类核心挑战
### 1. 网络不可靠
单机里函数调用几乎默认可靠；分布式系统里消息可能：

- 丢失。
- 乱序。
- 重复。
- 长时间延迟。
- 请求成功但应答丢失。

文章特别强调一个关键点：即使底层用了 TCP，也不能把“TCP 可靠”误解成“业务可靠送达”。TCP 只能提供传输层语义，不能替你判断业务是否真的被对端成功处理。

### 2. 时钟和顺序问题
单机系统里可以用统一物理时钟理解事件顺序；分布式系统里每台机器的时钟都可能漂移，网络延迟又不可预测，所以不能简单用物理时间戳判断事件先后。

对开发者来说，真正重要的是：

- 不要轻易把“时间戳更大”当成“事件更晚发生”。
- 要学会区分物理时间、逻辑顺序和因果关系。

### 3. 部分失效
这是文章里最值得反复理解的一点。单机系统常见的是“正常”或“宕机”的二态；分布式系统常见的是“不知道对方到底怎么了”的第三种状态。

例如一次扣款请求超时，调用方无法确定：

- 请求根本没到。
- 请求到了但服务崩了。
- 请求执行成功但响应丢了。
- 对方只是处理慢。

这类“不知道发生了什么”的状态，会引出：

- 重复执行。
- 数据不一致。
- 级联故障。
- 脑裂等幽灵行为。

### 4. 数据一致性
文章用副本延迟说明了一个核心事实：一旦数据复制到多节点，就不可能在所有时刻让所有节点瞬时保持相同视图。

因此一致性不是“有没有”的问题，而是“要多强、为此愿意牺牲多少延迟和可用性”的问题。

## 心智转变
文章最后提出了从单机到分布式最关键的四个转变。

### 从二态逻辑到三态逻辑
本地函数调用通常只有成功或失败；远程调用还会多一个“超时且未知结果”的状态。这直接要求系统具备 [[幂等性]]，并允许安全重试。

### 从全局一致到相对顺序
不能再依赖绝对时间，而要关注事件的顺序关系和因果关系。这是后续理解逻辑时钟、版本冲突和一致性模型的入口。

### 从完美主义到权衡艺术
分布式系统不存在免费午餐。文章用 CAP 的视角说明：当网络分区出现时，系统必须在一致性和可用性之间做取舍。

### 从防御故障到拥抱故障
在分布式系统里，故障不是异常，而是常态。真正成熟的系统设计不是假设不会出错，而是默认依赖会超时、节点会抖动、链路会中断，并提前设计熔断、隔离和降级。

## 对开发者有什么用
读完这篇文章，最直接的收益不是记住定义，而是以后在写服务调用、缓存更新、重试逻辑、分库分表、消息消费时，会更自然地问这几个问题：

- 如果请求超时，我到底知道了什么，不知道什么？
- 如果重试，这个操作会不会执行两次？
- 如果两个节点看到的数据不一样，业务能接受多久？
- 如果依赖服务很慢，会不会把我自己的线程池拖死？
