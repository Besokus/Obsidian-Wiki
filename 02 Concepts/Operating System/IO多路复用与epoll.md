# IO 多路复用与 epoll

## 问题背景

当需要同时监控多个 socket 时，如果为每个连接创建一个线程，在高并发下资源开销巨大。IO 多路复用允许单线程同时监控多个文件描述符。

## 它回答的核心问题
- 它试图解释：为什么高并发 socket 不能简单靠“一个连接一个线程”。
- 它试图约束：如何在一个线程里高效等待多个 fd 就绪。
- 它试图区分：轮询式多路复用和事件驱动式多路复用的成本差异。

---

## select / poll

本质是**轮询机制**，每次调用都需要把所有 fd 从用户态拷贝到内核态，并遍历检查是否就绪。

问题：
- O(n) 遍历，连接数多时开销大
- 频繁内核态/用户态拷贝
- select 有 fd 数量限制（默认 1024）

---

## epoll

epoll 是 Linux 下的高性能 IO 复用机制，通过**事件驱动**替代轮询。

### 核心数据结构

- **红黑树**：管理注册的 fd，增删改查 O(log n)
- **就绪链表**：存储已就绪的 fd，直接返回给用户

### 高效原因

1. **避免重复拷贝**：注册 fd 时只拷贝一次，之后不需要每次传入全部 fd
2. **事件驱动**：只返回就绪 fd，时间复杂度接近 O(1)
3. **无数量限制**：红黑树管理 fd，无上限
4. **就绪链表**：用户只处理真正有事件的连接

---

## 水平触发（LT）vs 边缘触发（ET）

| 对比 | LT | ET |
|---|---|---|
| 通知方式 | fd 就绪就一直返回，直到数据读完 | 仅状态变化时通知一次 |
| 通知次数 | 多 | 少 |
| 编程难度 | 低 | 高 |
| 性能 | 稍低 | 更高 |
| 是否丢数据 | 不会 | 可能（读不干净） |

### ET 使用要点

1. **必须将 fd 设置为非阻塞**
2. **一次事件处理中循环读/写直到返回 EAGAIN**
3. 否则会导致数据没读完但收不到下次通知 -> "假死"

### 实际选型

- **LT**：开发成本低，不出 bug，大部分中小并发足够
- **ET**：高并发场景（百万连接网关、直播服务器），需极致性能且能接受高编程复杂度

## 重点部分详解
- 机制细节：select/poll 是轮询和拷贝，epoll 是事件驱动和就绪队列。
- 常见故障：ET 下如果 fd 没设非阻塞，或者一次没读干净，就会出现“假死”。
- 典型代价：epoll 性能更高，但编码复杂度明显更大，尤其是 ET。
- 实战判断：连接数少、开发优先时选 LT；连接数大、性能敏感时再考虑 ET。

---

## Reactor 模式

以 epoll 为核心的事件驱动框架：主线程监听事件，事件到达时分发给工作线程处理。典型应用：Nginx、Redis。

## 和理论的关系
- 依赖的理论概念：事件驱动、就绪/阻塞、系统调用开销
- 关键假设：大部分时间 fd 都不会就绪，必须避免无意义轮询。
- 理论边界对工程的影响：IO 多路复用解决的是“等谁就绪”的问题，不解决业务处理耗时问题。
