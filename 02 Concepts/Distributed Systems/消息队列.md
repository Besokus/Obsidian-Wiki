# 消息队列

## 1. 为什么需要消息队列

消息队列（Message Queue）的核心作用是在**生产者**和**消费者**之间通过**异步中间层**解耦通信。

主要解决三类问题：

```text
解耦：生产者不需要知道谁在消费
削峰：缓冲瞬时高流量，防止下游被冲垮
异步：不需要立即返回结果的操作可以异步处理
```

### 典型场景

- 订单创建后发送通知（邮件、短信、推送）
- 秒杀流量削峰，订单异步落库
- 微服务间事件通知
- 日志和业务事件采集
- 数据管道和流处理

---

## 2. 主流 MQ 对比：Kafka vs RabbitMQ

| 特性 | RabbitMQ | Kafka |
|---|---|---|
| 类型 | 传统消息队列（Broker + Queue） | 分布式日志系统（Commit Log） |
| 设计目标 | 可靠传输、灵活路由 | 高吞吐、持久化、流式处理 |
| 消息模型 | Queue，消费后默认删除 | Partition + Offset，消息保留可配置 |
| 消费方式 | Push / Pull | Pull（通过 `poll()`） |
| 消息顺序 | 单队列内顺序 | 分区内顺序，分区间无全局顺序 |
| 路由能力 | 灵活（Exchange + Binding） | 弱（Topic + Partition） |
| 吞吐量 | 中等 | 高 |
| 消息保留 | 消费后通常删除 | 可长期保留，支持回溯 |
| 生态定位 | 业务解耦、任务队列 | 大数据管道、流处理、日志采集 |

### 选型建议

```text
业务解耦、复杂路由、低延迟任务   → RabbitMQ
高吞吐、日志管道、流式数据处理    → Kafka
简单异步任务、小规模消息          → Redis Stream 也可考虑
```

## 3. Kafka 核心机制

### 3.1 基本架构

```text
Producer
  ↓
Topic (Partition 0 / Partition 1 / Partition 2 ...)
  ↓
Consumer Group (多个 Consumer)
```

### 3.2 Pull 模型

Kafka 采用 Pull 模型，消费者通过 `poll()` 主动拉取消息，而非服务端推送。

优点：
- 消费者控制消费速率
- 支持批量拉取，提升吞吐
- 不会压垮消费者

### 3.3 Partition 与 Consumer Group

- 一个 Topic 分为多个 Partition，每个 Partition 内消息有序
- 同一 Consumer Group 内，一个 Partition 只分配给一个 Consumer
- Consumer 加入/退出时触发 **Rebalance**，重新分配 Partition

### 3.4 Offset 管理

- Kafka 用 Offset 记录消费进度
- 消费者处理完消息后提交 Offset
- 故障恢复后从上次 Offset 继续消费
- 配合幂等，避免重复消费

## 4. 消息可靠性

消息丢失的风险存在于三个环节，需要分层防护：

```text
生产者 → Broker → 消费者
```

### 4.1 生产者阶段：发送确认

| 级别 | 行为 | 风险 |
|---|---|---|
| acks=0 | 不等待确认 | 可能丢 |
| acks=1 | Leader 确认即返回 | Leader 宕机可能丢 |
| acks=all | 所有副本确认 | 最可靠 |

生产端应配置**发送确认 + 重试机制**。

### 4.2 Broker 阶段：持久化与副本

- 消息持久化到磁盘（Kafka 利用顺序写，性能高）
- 多副本机制，副本同步后才确认
- 避免单点故障导致数据丢失

### 4.3 消费者阶段：消费确认

- 处理完业务逻辑后再提交 ACK / Offset
- 避免"先确认后处理"导致消息丢失
- 配合重试和死信队列处理失败消息

### 4.4 关于"丢消息"的常见问答

> Q: MQ 消息丢了怎么办？

A: 从三个阶段防护：

1. **生产者**：开启确认机制（acks=all），发送失败重试
2. **Broker**：持久化磁盘，多副本同步
3. **消费者**：业务处理完再确认，不先 ack 后处理

如果对可靠性要求极高，结合**本地消息表**或**事务性发件箱（Transactional Outbox）** 确保消息必定投递。

## 5. 重复消费与幂等

### 问题

MQ 的"至少一次"语义（at-least-once）下，重复消费是常态：

```text
消费者处理完但 ack 超时
消费者宕机重启后重新拉取
Kafka Rebalance 后重复消费
```

### 解决方案：幂等

核心思路是**接口幂等**，让重复执行和一次执行结果相同：

```text
全局唯一 ID（消息 ID / 业务 ID）
  ↓
消费者判断是否已处理
  ↓
已处理 → 直接 ACK，不重复执行
未处理 → 执行业务逻辑，记录处理状态
```

常用去重方案：

- 数据库唯一键约束（如业务 ID 做唯一键）
- Redis 去重（记录已处理的消息 ID，TTL 设置合理）
- 业务逻辑本身就是幂等的（如 `set` 操作）

## 6. 死信队列（DLQ）

### 什么是死信队列

死信队列（Dead Letter Queue）是用于存放"无法被正常消费的消息"的专用队列。

### 消息进入死信队列的常见原因

1. 消费失败超过最大重试次数
2. 消息过期（TTL 到期）
3. 队列长度达到上限
4. 消费者显式拒绝且不重回队列

### 死信队列的作用

```text
防止消息无限重试 → 保护主链路，防止线程占满
提供问题排查入口 → 错误消息保留，用于分析
支持补偿与人工干预 → 修复 bug 后可重新投递
```

### 工程建议

- 监控死信队列数量和内容
- 设定报警规则，死信过多时及时介入
- 保证死信队列本身的消息不丢失

---

## 7. 工程综合建议

实际系统中 MQ 的**最常问问题**可以概括为四个维度：

| 问题 | 核心策略 |
|---|---|
| 消息丢了 | 分层防护：生产确认 + 持久化 + 消费确认 |
| 重复消费 | 幂等：唯一 ID + 去重 + 业务幂等 |
| 消息积压 | 扩容 Partition + 提升消费能力 + 限流 |
| 顺序消费 | 分区内有序（Kafka）/ 单队列有序（RabbitMQ） |
