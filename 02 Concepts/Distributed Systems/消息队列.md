# 消息队列

## 一句话解释
消息队列（MQ）是在生产者和消费者之间增加的异步中间层，用来解耦服务、削峰填谷、异步处理和广播事件。

## 它解决什么问题
- 正确性：避免调用链过长时，一个下游失败直接把整个主流程拖死。
- 性能：把“必须同步返回”的操作和“可以稍后处理”的操作分开。
- 可用性：下游短时故障时，消息可以先堆在队列里。
- 可维护性：生产者不需要知道所有消费者是谁，系统演化更容易。

## 先定边界
- 它是协议 / 模式 / 组件 / 机制 / 策略 / 工具中的哪一种？  
  它首先是**基础设施组件**，背后带一套消息投递和消费语义。
- 它主要解决的是正确性、性能、可用性、扩展性还是可维护性问题？  
  它主要解决**解耦、异步、削峰和事件分发**问题，同时引入可靠性治理问题。

## 什么时候用
- 订单创建后通知短信、邮件、积分、搜索索引、推荐系统。
- 秒杀削峰，把同步流量转成异步处理。
- 事件驱动架构中的领域事件广播。
- 日志采集、埋点上报、数据管道、流式处理。

## 什么时候不用
- 调用方必须立刻拿到结果，并且强依赖下游执行结果。
- 业务无法接受异步延迟和最终一致。
- 数据量很小、链路极短，引入 MQ 只会增加运维和复杂度。

## 核心机制

```text
1. Producer 把消息写入 Broker
2. Broker 按 queue / topic / partition 持久化和分发消息
3. Consumer 拉取或接收消息
4. Consumer 处理业务并提交 ack / offset
5. 失败时进入重试、死信、补偿或人工处理链路
```

复盘时最该记住的重点：
- MQ 不是“可靠传输黑盒”，而是“生产、存储、消费”三段式系统。
- 每一段都可能丢、重、乱、慢，所以每一段都要单独治理。

## 开发者视角
- 常见 API / 配置：Kafka 的 `acks`、`retries`、`consumer group`、`offset`；RabbitMQ 的 exchange、routing key、ack、DLQ。
- 关键参数：分区数、消费并发、批量大小、重试次数、保留时间、死信策略。
- 失败模式：消息丢失、重复消费、消息积压、顺序被打散、Rebalance 抖动、消费者雪崩。
- 性能影响：分区数影响并行度，持久化和副本数影响延迟，批量拉取影响吞吐。
- 并发 / 线程安全：同一 partition 通常只保证单分区内顺序；多消费者并发会天然打散全局顺序。
- 观测与排查：重点盯生产失败率、消费 lag、重试率、死信量、Rebalance 次数、积压时长。

## 工程判断
- 判断信号：主流程被非关键下游拖慢；同步链路过长；流量高峰把下游打崩；需要事件广播。
- 关键权衡：解耦与吞吐换来的是最终一致、重复消费、排障复杂度。
- 常见替代方案：[[RPC]]、定时任务、批处理、[[Outbox Pattern]]、[[Transactional Message]]。

## 最小例子

```text
用户下单成功
-> 订单服务同步返回“下单成功”
-> 异步发 MQ：
   - 积分服务加积分
   - 搜索服务更新索引
   - 通知服务发短信
```

这里最关键的判断是：这些下游任务是否必须和下单一起同步完成。如果不是，MQ 就比 RPC 更合适。

## 实际故障 / 代价
- 常见故障：消息丢失、重复消费、消费积压、顺序错乱、Broker 故障、消费者 Rebalance 抖动。
- 错误使用方式：把 MQ 当强事务；认为“有 MQ 就绝不会丢”；消费逻辑不幂等；重试无上限。
- 典型代价：链路延迟不可控、问题更分散、排障要跨 producer/broker/consumer 三层。

## 重点部分详解

### 1. 可靠性重点：消息为什么会丢

```text
Producer -> Broker -> Consumer
```

三段都可能出问题：
- Producer 发出去了但没确认
- Broker 没持久化或副本没同步
- Consumer 先 ack 后处理，业务还没落地就把消息“吃掉了”

复盘重点：不要笼统问“MQ 会不会丢”，要问“丢在生产、存储还是消费阶段”。

### 2. 重复消费重点：为什么它是常态
只要系统追求 at-least-once，就必须接受：

```text
处理成功但 ack 丢了
消费者重启后重复拉取
Rebalance 后重新消费
```

所以 MQ 的正确打开方式不是“避免重复”，而是“接受重复并让消费幂等”。参见 [[幂等性]]。

### 3. 顺序重点：局部有序，不等于全局有序
- Kafka：通常是 **partition 内有序**
- RabbitMQ：通常是 **单 queue 内近似有序**

只要你做多分区、多消费者并发，就不能再默认全局顺序。

### 4. 削峰重点：MQ 只能缓冲，不能消灭流量
MQ 可以把峰值流量变成可排队流量，但如果消费能力长期跟不上，积压只会越来越大。削峰必须和扩容、限流、降级一起看。

## 常见误解
- “上了 MQ，系统就解耦了”：只是通信解耦，数据语义和故障恢复仍然耦合。
- “MQ 天然可靠”：可靠性是配置、协议和工程治理共同实现的。
- “有分区就一定高吞吐”：分区数、热点 key、消费能力、下游数据库都可能成为新瓶颈。

## 易混概念
- [[RPC]]：同步调用，强调立即拿结果；MQ 是异步投递，强调解耦和缓冲。
- [[Outbox Pattern]]：解决“本地事务 + 发消息不原子”；MQ 本身不自动解决这个问题。

## 和理论的关系
- 依赖的理论概念：[[最终一致性]]、[[幂等性]]、[[CAP]]
- 关键假设：系统能接受异步延迟、重复投递和最终一致。
- 理论边界对工程的影响：如果业务要求写后立刻全局可见，MQ 往往不是主路径协调机制。

## 我的理解
消息队列最大的价值不是“把数据搬过去”，而是把系统从“同步依赖链”改造成“异步协作网络”。真正难的不是接入 MQ，而是接入之后承认并治理它带来的重复、延迟、积压和最终一致。
