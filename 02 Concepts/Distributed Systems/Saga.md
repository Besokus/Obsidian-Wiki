# Saga（本地事务 + 补偿）

## 一句话解释
Saga 把一个长事务拆成一串本地事务 `T1..Tn`；当中途失败时，用补偿事务 `Ck` 抵消已经发生的业务效果，以达到“最终一致、可解释”的结果。

> Saga 的补偿不是数据库 rollback，而是业务层反向动作（参见 [[补偿]]）。

---

# 1. 两种实现方式

## 1.1 编排式（Orchestration）
由一个 orchestrator 驱动步骤推进与失败补偿。
- 优点：流程可控、易观测、易治理（重试/超时/告警集中）
- 缺点：协调者复杂度与可用性要求更高（需要 [[事务状态机]]）

## 1.2 协同式（Choreography）
服务通过事件协作推进（A 发事件 → B 做事 → 再发事件…）。
- 优点：解耦、扩展性好
- 缺点：排查与全局治理更难，容易“事件风暴”

---

# 2. Saga 的关键代价：弱隔离 + 中间态
Saga 通常不提供强隔离：执行过程中可能出现“部分步骤已完成”的中间态。
工程上需要：
- 设计用户可见状态（处理中/待确认/失败可重试）
- 配套 [[对账]] 与补偿任务
- 所有动作与补偿都要求 [[幂等性]]

---

# 3. 什么时候用（工程判断）
更合适：
- 微服务长流程（下单 → 支付 → 发货 → 积分…）
- 能接受最终一致与中间态

不太合适：
- 需要强隔离/强一致的短链路（倾向 [[TCC]] / [[2PC]]）
- 补偿不可行或代价巨大（不可逆动作过多）
