# 因果一致性

## 一句话解释
因果一致性要求有因果关系的写入必须按因果顺序被观察到；没有因果关系的写入可以以不同顺序被不同客户端观察到。

## 它解决什么问题
最终一致不保证顺序，线性一致则对一切操作排序（成本高）。因果一致性只对"真正有关联"的操作排序，既保证了业务语义的合理性，又避免了全局排序的高昂代价。

- 正确性：用户不会看到违反因果关系的状态（如看到评论却看不到原始帖子）。
- 性能：允许并发无因果关系的操作并行，不需要全局同步。
- 可用性：比强一致的可用性边界更宽。

## 什么时候用
- 社交网络：先发帖、后评论、再回复——任何用户不应看到回复但没有帖子。
- 协作文档、聊天系统、消息流。
- 任何用户行为之间存在明确因果依赖的场景。

## 什么时候不用
- 没有明显因果依赖的纯数值型操作（计数、累加）——最终一致足够。
- 需要严格实时可见的判断逻辑（余额、锁）——需要线性一致。
- 业务不关心观察顺序——最终一致更简单。

## 核心机制
因果关系通常来自三方面：
```text
1. 同一客户端的程序顺序：发帖在前，评论在后。
2. 读后写：读到某人的帖子后再评论，评论"依赖于"帖子。
3. happens-before 关系：通过向量时钟或版本号传递因果依赖。
```

实现上通常依赖向量时钟的变体或显式的依赖传播机制。

## 开发者视角
- 因果一致性比最终一致多了"顺序不乱"的承诺，是工程上性价比很高的一种模型。
- 实现因果一致需要追踪依赖关系（向量时钟是经典但并非唯一的方案）。
- 因果一致不能解决"冲突"——两个无因果关系的写入如果写同一个 key，仍然可能冲突。

## 工程判断
- 判断信号：业务场景中是否存在明确的"先因后果"依赖（发帖→评论、建商品→改价格）。
- 关键权衡：比最终一致多了一个"因果关系正确"的保证，但实现复杂度增加。
- 常见替代方案：会话一致性（同一用户维度的顺序保证）、强一致（如果数据量小且不需要高可用）。

## 最小例子
```text
Client A: write post "hello" → 成功
Client A: write comment "nice post" on that post
Client B: reads → 绝不允许先看到评论后看到帖子
```

## 常见误解
- "因果一致就是顺序一致"：不对。因果一致只约束有因果关系的操作，并发无关的操作可以任意排序。
- "因果一致很贵"：它通常比线性一致便宜得多，但比最终一致复杂。

## 易混概念
- [[线性一致性]]：所有操作按真实时间排序，包括无因果关系的操作。
- [[顺序一致性]]：所有操作按一个全局序列排序，但不依赖真实时间。
- [[最终一致性]]：不保证任何顺序，只保证收敛。

## 我的理解
因果一致性是分布式系统中"性价比"很高的一个模型——它为需要顺序的地方提供顺序，为不需要的地方保留并发。在社交网络、评论系统、协作文档这类业务中，因果关系就是业务的自然边界。
