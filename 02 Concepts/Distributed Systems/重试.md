# 重试

## 一句话解释
重试是在一次调用失败后再次尝试的机制，用来提高“短暂性故障”下的成功率；但不受控的重试会放大负载并触发雪崩。

## 前置知识
理解它之前最好先知道什么？

- [[超时]]
- [[幂等性]]
- [[网络不可靠]]

## 它解决什么问题
网络抖动、短暂过载、瞬时连接失败、主从切换窗口等“可恢复失败”，可能通过一次或少量重试成功。

- 正确性：前提是幂等或有去重，否则会带来重复执行风险。
- 性能：重试会增加额外请求，必须可控。
- 可用性：正确重试能提升成功率；错误重试会降低可用性。
- 可维护性：需要区分可重试/不可重试错误并可观测。

## 什么时候用
- 读请求或幂等写请求。
- 明确的可恢复错误：超时、连接重置、429/503 等（前提是下游用这些状态表达“请退避”）。
- 重试次数少、延迟预算允许、且有退避策略。

## 什么时候不用
- 非幂等写（扣款、发券、发短信、创建资源）且没有幂等键/去重机制。
- 下游持续失败或明显过载时的盲目重试（会形成重试风暴）。
- 多层都各自重试且没有统一预算（会指数放大）。

## 重试策略
| 策略 | 做法 | 适用 |
|---|---|---|
| 固定间隔 | 每隔固定时间重试 | 简单场景，但容易同步冲击 |
| 指数退避 | 间隔按 2^n 增长 | 多数远程调用 |
| 指数退避 + 抖动 | 间隔随机化 | 推荐方式，避免集中重试 |

## 核心机制
```text
1. 先设置整体超时 + 单次尝试超时（per-try timeout）。
2. 只对”可重试”的错误重试。
3. 用指数退避（exponential backoff）。
4. 加抖动（jitter）打散重试波峰。
5. 限制每个请求的最大重试次数，并设置进程/实例级 retry budget。
```

多层无控制重试的放大效应：网关、服务 A、服务 B 各自重试 2 次，最坏情况下一次请求可能被放大为 3×3×3 = 27 次调用（重试风暴）。

## 重点部分详解

### 1. 为什么重试能成功
很多失败不是永久性错误，而是短暂抖动、切换窗口、瞬时过载。等一会儿再试，成功率会明显上升。

### 2. 为什么重试会放大故障
如果失败本身就是过载引起的，你再重试就等于继续加压。没有预算的重试会把局部故障放大成全局故障。

### 3. 为什么必须配合抖动
没有 jitter 时，多个请求会在同一时间窗口一起重试，形成新的尖峰。

### 4. 为什么必须配合幂等
重试允许同一个动作被执行多次。没有幂等，就会出现重复扣款、重复发货等副作用。

## 开发者视角
- 重试和超时必须配套：没有超时的重试会让资源泄漏。
- 退避 + 抖动是关键，否则重试会在同一时间窗口集中爆发。
- 避免多层重试叠加：只在一个“合适的层”重试，或使用统一预算与统一策略。
- 需要区分错误类型：业务失败（例如参数错误）不应重试。

## 工程判断
遇到真实问题时，如何判断是否该用它？需要看哪些证据？

- 判断信号：失败类型是否可恢复、失败比例是否低、是否存在明显的短暂尖峰。
- 关键权衡：提升成功率 vs 放大负载；延迟更高 vs 更少失败。
- 常见替代方案：排队/异步化、缓存、幂等+补偿、限流/负载卸载、熔断。

## 最小例子
一个常见且有效的重试形态：

```text
max_attempts = 3
backoff = exponential + jitter
only retry on: timeout/connection reset/429/503
per_try_timeout < overall_timeout
```

## 常见误解
- “失败就重试”：持续失败时重试会把故障放大成雪崩。
- “多重试几次就稳了”：没有退避/抖动/预算的重试只是在制造负载。

## 易混概念
- [[幂等性]]：重试安全性的前提。
- [[熔断]]：熔断解决“持续失败仍在打”，重试解决“短暂失败再试一次”。
- [[降级]]：重试失败后必须有 fallback，否则只是延迟更高的失败。

## 我的理解
重试不是“容错”，而是“用额外负载换取更高成功率”的交易。做对的标志是：失败时系统更稳，做错的标志是：失败时 QPS 更高、延迟更长、故障更大。
