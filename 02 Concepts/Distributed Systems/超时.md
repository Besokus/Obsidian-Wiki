# 超时

## 一句话解释
超时是调用方对一次远程调用“最多等多久”的硬上限。它把“可能永远不返回”的网络调用，变成一个可控的失败模式。

## 前置知识
理解它之前最好先知道什么？

- [[网络不可靠]]
- [[部分失效]]
- [[RPC]]

## 它解决什么问题
没有超时，调用链会把线程、连接、内存等资源无限期占住，慢依赖会演化成线程池/连接池耗尽，进一步触发级联故障。

- 正确性：超时不会直接提升正确性，但能避免“系统卡死”导致的连锁错误。
- 性能：限制尾延迟对资源占用的放大效应。
- 可用性：让系统在依赖变慢时仍能“快速失败并恢复”。
- 可维护性：为重试、熔断、降级提供清晰边界。

## 什么时候用
- 所有外部依赖调用都要有超时：RPC/HTTP、数据库、缓存、消息系统、第三方接口。
- 有明显尾延迟（P95/P99）问题的接口，需要用超时控制资源占用。
- 需要全链路延迟预算（deadline budget）的请求链路。

## 什么时候不用
- 基于流式响应/长连接的场景，不适合用“请求总超时”，更适合用 `idle timeout`（空闲超时）或心跳机制。

## 常见超时类型
- 连接超时（connect timeout）：建立连接的最大等待时间。
- 读取超时（read / socket timeout）：连接建立后等待响应数据的最大时间。
- 请求总超时（request timeout / deadline）：整个请求从开始到结束的最大耗时。在全链路场景中比单点超时更重要，通常使用 deadline 传播。

## 核心机制
```text
1. 入口设置总 deadline（总预算）。
2. 向下游传播 deadline（或剩余预算）。
3. 每次调用都设置 per-try timeout（单次尝试上限）。
4. 超时触发后取消/中断下游工作（cancellation propagation）。
```

## 开发者视角
- 不设置超时几乎等价于“把线程池交给未知延迟”。
- 超时不是拍脑袋，通常基于历史延迟分布（P95/P99）+ 冗余预算设置。
- 上游超时应覆盖下游（含重试）但不能无限放大，否则上游早已放弃，下游还在做无意义工作。
- 有重试时应使用 `per_try_timeout` + `overall_timeout` 的组合，避免单次尝试吃光总预算。

## 工程判断
遇到真实问题时，如何判断是否该用它？需要看哪些证据？

- 判断信号：线程池/连接池占用升高、in-flight 请求数飙升、尾延迟拉长、下游超时率上升。
- 关键权衡：超时过短会误伤（假失败），过长会资源被拖死（真雪崩）。
- 常见替代方案：并行请求（hedged requests）、缓存、异步化、隔离/限流，但这些不替代超时本身。

## 最小例子
把“入口总预算”往下传，而不是每一层都用更长超时：

```text
入口总预算 1000ms
服务 A 预算 300ms
服务 B 预算 250ms
DB 预算 200ms
剩余用于序列化/排队/网络
```

## 常见误解
- “用了 TCP 就不需要超时”：TCP 不保证业务层完成，也不能避免长尾延迟。
- “超时就是失败证明”：超时只是“我不等了”，不等于对方没执行/没成功。

## 易混概念
- [[重试]]：重试是动作，超时是边界；无超时的重试会把资源拖死。
- [[熔断]]：熔断解决“持续失败还继续打”的问题，超时解决“单次等待太久”的问题。
- [[降级]]：超时触发后通常需要降级策略来保住核心体验。

## 我的理解
超时是稳定性治理里最基础、也最常被忽略的“硬约束”。它不让调用链用不确定的时间占用确定的资源，这是避免级联故障的第一步。
