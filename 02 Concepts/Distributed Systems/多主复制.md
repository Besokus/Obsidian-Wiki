# 多主复制

## 一句话解释
多主复制中，多个节点都可以接收写请求并互相同步。它主要用于跨数据中心场景，让每个地域都能本地写入以降低延迟，但代价是必须处理并发写入冲突。

## 它解决什么问题
主从复制要求所有写请求打到同一个 Leader，跨地域时写入延迟很高（每次写要跨洋）。多主复制让每个数据中心都有自己的 Leader，本地写入、异步互相同步。

- 延迟：跨地域写入不需要跨洋往返。
- 可用性：某个数据中心故障时，其他数据中心仍可接收写入。
- 复杂性：写冲突、冲突解决、调试难度都显著增加。

## 什么时候用
- 跨地域数据中心，用户分布在多个大洲。
- 需要离线工作的系统（移动端本地写入后同步）。
- 写入延迟敏感、不能接受跨地域往返。

## 什么时候不用
- 写冲突无法合理解决（如严格递增的余额操作）。
- 业务无法接受"最终收敛"的不确定窗口。
- 团队没有处理冲突和复现线上问题的能力——多主复制的调试成本很高。

## 核心机制
```text
1. 每个数据中心有自己的 Leader，接受本地写请求。
2. Leader 之间异步互相同步变更。
3. 冲突发生时，按预定策略解决。
```

## 写冲突
多主复制的核心问题是并发写入同一数据：

```
US Leader: user.name = "Alice"
EU Leader: user.name = "Alicia"
```

两个写入都在本地成功，但同步时发生冲突。

### 冲突解决策略

| 策略 | 做法 | 适用 |
|---|---|---|
| Last Write Wins | 按时间戳选最新值 | 简单场景，可能丢更新 |
| 固定优先级 | 某数据中心/节点优先 | 主备多主 |
| 业务合并 | 应用逻辑合并冲突值 | 购物车、列表类数据 |
| 保留多版本 | 让应用层或用户后续处理 | 需要精确语义的场景 |
| CRDT | 数据结构保证自动收敛 | 计数器、集合、文档协作 |

## 开发者视角
- 尽量避免需要强一致的写操作使用多主复制。大部分业务适合：用户修改自己的数据，天然按用户 ID 隔离，冲突概率低。
- 冲突解决要尽早做（写入时），不要推迟到读取时——否则系统行为难以预测。
- Last Write Wins 最简单但依赖时钟，时钟偏差可能导致"旧值覆盖新值"。

## 常见误解
- "多主比单主更高级"：只是在不同维度做取舍。多主解决的是跨地域写入延迟，不是更好的性能或更强的容错。
- "冲突很少所以没关系"：即使概率低，也需要明确冲突发生时系统的行为。

## 易混概念
- [[主从复制]]：单 Leader 接收写，冲突少但跨地域写入延迟高。
- [[无主复制]]：没有 Leader 概念，客户端直接多写，与多主的"多写入点共享"有相似之处但架构不同。

## 我的理解
多主复制解决了一个实在问题——跨地域写入延迟，但为此付出的冲突处理和运维复杂度不可低估。选择它的前提是：你的数据模型天然冲突少（如用户隔离），且你的团队能驾驭"最终收敛"的语义。
