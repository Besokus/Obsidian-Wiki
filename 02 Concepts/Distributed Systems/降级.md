# 降级

## 一句话解释
降级是在依赖不可用、系统过载或资源紧张时，主动返回“低质量但可用”的结果，保住核心链路和用户关键体验。

## 前置知识

- [[超时]]

## 它解决什么问题
当依赖失败时，如果只能“报错/卡住”，系统要么不可用，要么把资源耗尽。降级的目标是：

- 让核心功能继续可用（下单/支付/登录等）。
- 让非核心功能变差但不致命（推荐/评论/排行等）。
- 在过载时通过“更便宜的处理”降低资源消耗，避免雪崩。

## 什么时候用
- 非核心依赖失败：返回默认值、缓存值、简化结果。
- 系统过载：关闭昂贵功能、降低计算开销、减少下游调用。
- 灾备模式：只保留核心流程，非核心模块全部静默降级。

## 什么时候不用
- 会破坏业务正确性的场景（例如把支付结果降级成“成功”）。
- 降级结果无法被业务接受，或会引发更严重的后续错误。

## 降级分类
- 自动降级：由系统根据指标（错误率、延迟、线程池耗尽等）自动触发。
- 手动降级：运维/研发通过开关关闭非核心功能，常见于大促期间。
- 预案降级：提前定义好降级方案（如支付渠道 A 不可用时切换到渠道 B）。

## 核心机制
```text
1. 识别核心/重要/非核心功能分级。
2. 为每个依赖设计 fallback（缓存/默认值/近似值/延迟加载）。
3. 在超时/熔断/过载条件触发时启用降级。
4. 记录降级次数与命中率，定期演练。
```

## 重点部分详解

### 1. 为什么要分级
不是所有功能都同等重要。真正要保的是用户关键路径，次要功能变差通常可以接受。

### 2. 为什么 fallback 必须合法
一个“空值”如果会把前端或下游搞崩，那就不是 fallback，而是另外一种故障。

### 3. 为什么降级要和熔断一起看
熔断告诉系统“别再打了”，降级告诉系统“那给什么结果”。两者缺一不可。

### 4. 为什么要演练
很多 fallback 只有在故障时才暴露出它本身也依赖同一故障组件。

## 开发者视角
- 降级方案必须“真的能跑”，否则故障时会发现 fallback 也依赖同一故障组件。
- 降级要可观测：降级率、触发原因、对核心指标影响。
- 降级常和“负载卸载/丢弃”一起出现：必要时宁可拒绝部分请求也不要拖垮全部。

## 工程判断
遇到真实问题时，如何判断是否该用它？需要看哪些证据？

- 判断信号：关键依赖失败/超时率升高、线程池/队列接近饱和、P99 拉长。
- 关键权衡：结果质量下降 vs 整体可用性提升；一致性/新鲜度 vs 延迟。
- 常见替代方案：扩容、限流、隔离、异步化；但这些不替代“功能分级 + fallback”。

## 最小例子
商品详情页：

```text
商品信息：必须成功，否则报错页
价格/库存：重要，允许缓存兜底
推荐：非核心，失败返回默认推荐或热门榜
```

## 常见误解
- “有 fallback 就够了”：没有演练和监控的 fallback 等于没有。
- “降级就是返回空”：空值可能破坏前端/调用方假设，默认值也要可用且合法。

## 易混概念
- [[熔断]]：熔断决定“还打不打”，降级决定“失败后给什么”。
- [[重试]]：重试失败后通常需要降级，否则只是更慢的失败。
- [[超时]]：超时触发的失败需要降级兜底才能保体验。

## 我的理解
降级不是“认输”，而是把系统目标从“全功能”切换为“核心可用”。能不能降级取决于你是否提前做了功能分级，并准备了可运行的 fallback。
