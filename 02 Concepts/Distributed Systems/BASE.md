# BASE

## 一句话解释
BASE 是面向高可用分布式系统的一种工程思路：允许短暂不一致，靠异步同步、修复和收敛机制最终达到一致。

## 它回答的核心问题
- 它试图解释：为什么分布式系统不一定要追求任意时刻都强一致。
- 它试图约束：系统如何在可用性优先时维持可接受的正确性。
- 它试图区分：BASE 不是协议，而是一组设计取向。

## 它解决什么问题
- 让系统在故障或高压下仍然能对外服务。
- 把强一致的同步成本拆成异步收敛成本。
- 为缓存、消息队列、复制、补偿提供统一设计思路。

## 重点部分详解
### Basically Available
- 系统出故障时尽量还能对外提供部分功能。
- 常见表现是限流、降级、部分接口不可用但核心链路可用。

### Soft State
- 系统状态允许短时间内不稳定。
- 复制数据、缓存、派生索引等可在后台渐进修复。

### Eventually Consistent
- 如果没有新的写入，系统最终会收敛到一致状态。
- 重点不是“现在一致”，而是“是否有明确的收敛机制”。

## 核心机制
```text
1. 把强同步链路拆成异步流程。
2. 允许中间态短暂不一致。
3. 通过重试、补偿、对账、修复、版本控制收敛。
4. 为高可用保留降级和兜底能力。
```

## 常见工程机制
- [[消息队列]]：把同步链路拆成异步处理。
- [[幂等性]]：允许重复投递或重复消费。
- [[重试]]：处理短暂失败。
- [[补偿]]：修正已发生的错误状态。
- [[对账]]：周期性发现和修复不一致。
- [[事务状态机]]：用显式状态减少半成功问题。

## 易混概念
- [[ACID]]：更偏强事务语义。
- [[CAP]]：更强调分区时的权衡。
- [[PACELC]]：把延迟也纳入取舍。

## 我的理解
BASE 的核心不是“松”，而是“把一致性拆成工程过程”。它适合高可用系统，但前提是你真的准备好了收敛、补偿和监控。
