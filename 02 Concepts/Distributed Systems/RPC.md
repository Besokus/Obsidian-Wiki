# RPC

## 一句话解释
RPC（Remote Procedure Call，远程过程调用）是一种让程序像调用本地方法一样调用远程服务方法的通信方式。它屏蔽了底层网络传输、序列化、连接管理和响应解析等细节。

## 它解决什么问题
在分布式系统中，服务通常部署在不同进程、不同机器甚至不同机房。一个服务想调用另一个服务的能力时，不能直接访问对方内存里的函数，只能通过网络通信完成。

RPC 主要解决这些问题：

- 调用抽象：让远程调用看起来接近本地方法调用。
- 网络细节封装：调用方不需要手写 Socket、连接管理、请求编码和响应解析。
- 数据传输：把方法名、参数和返回值转换成可跨网络传输的格式。
- 服务协作：让多个服务可以通过明确接口组合成更大的系统。

## 什么时候用
- 微服务之间需要同步调用。
- 客户端希望调用服务端暴露的明确方法接口。
- 内部服务之间追求强类型接口、较高性能和清晰契约。
- 调用模式更像“请求一个方法并等待结果”，而不是事件流或广播。

## 什么时候不用
- 调用方不需要立即得到结果时，消息队列可能更合适。
- 服务之间耦合需要尽量降低时，事件驱动可能更合适。
- 跨公网、开放 API 或浏览器调用场景，HTTP REST/GraphQL 可能更通用。
- 业务无法接受远程调用失败、超时和重试带来的复杂性时，不应该把本地调用思维直接套到 RPC 上。

## 核心机制
一次典型 RPC 调用可以拆成以下流程：

```text
1. 服务消费端像调用本地方法一样发起调用。
2. 客户端 Stub 拦截调用，组装类名、方法名、参数等信息。
3. 请求对象被序列化成可网络传输的消息，例如 RpcRequest。
4. 客户端通过网络把请求发送到服务提供端。
5. 服务端 Stub 接收请求并反序列化。
6. 服务端根据类名、方法名和参数调用本地真实方法。
7. 服务端把执行结果封装成响应对象，例如 RpcResponse。
8. 响应被序列化后发回客户端。
9. 客户端反序列化响应，得到最终返回值。
```

## 重点部分详解

### 1. RPC 的本质错觉
RPC 只是把远程调用伪装成本地调用，本质还是网络通信。它不会消除超时、失败、重复和抖动，只是把这些复杂性封装起来。

### 2. 重试与幂等
RPC 一旦加入重试，就可能把一次调用变成多次调用，因此服务端必须考虑 [[幂等性]]。非幂等接口不能盲目重试。

### 3. 序列化与兼容性
RPC 不只是传输，还涉及消息格式、schema、版本演进和跨语言能力。协议选型会直接影响长期维护成本。

### 4. 同步链路的代价
RPC 会把调用链串起来，链路越深，尾延迟和故障传播越明显。很多微服务事故不是单点故障，而是 RPC 链太长。

## 关键组件
| 组件 | 作用 |
|---|---|
| Client | 服务消费端，发起远程调用 |
| Client Stub | 客户端代理，负责把本地方法调用转换成网络请求 |
| Transport | 网络传输层，例如 Socket、Netty、HTTP/2 |
| Server Stub | 服务端请求分发层，负责解析请求并调用真实方法 |
| Server | 服务提供端，真正执行业务逻辑 |
| Serializer | 序列化协议，例如 JSON、Protobuf |

## 开发者视角
- RPC 看起来像本地调用，但本质上是网络调用，必须考虑超时、失败、重试和幂等。
- 方法参数和返回值需要序列化，序列化协议会影响性能、兼容性和跨语言能力。
- 服务地址需要发现和管理，真实框架通常还会涉及注册中心、负载均衡和连接池。
- 重试可能导致服务端重复执行，因此写接口时要考虑 [[幂等性]]。
- RPC 会放大服务之间的同步依赖，调用链过长会增加延迟和故障传播风险。

## 常见框架
### gRPC
gRPC 是 Google 开源的高性能 RPC 框架，基于 HTTP/2，并通常使用 Protocol Buffers 作为接口定义和序列化协议。

它的特点：

- 支持多语言。
- 支持 HTTP/2 的双向流、头部压缩和多路复用。
- 使用 `.proto` 文件定义服务接口和消息结构。
- 适合内部服务通信、跨语言服务调用和高性能 RPC 场景。

### Protocol Buffers
Protocol Buffers 是一种跨语言、跨平台的数据序列化协议。它比纯文本格式更紧凑，适合网络通信和数据存储，但需要提前定义 schema。

## 易混概念
- [[HTTP]]：HTTP 是应用层协议，RPC 是调用模型；RPC 可以基于 HTTP/2、TCP 等传输实现。
- [[消息队列]]：RPC 偏同步请求响应，消息队列偏异步解耦。
- [[服务发现]]：RPC 框架通常需要知道服务地址，服务发现负责找到可用实例。
- [[幂等性]]：RPC 重试时避免重复执行副作用的关键能力。

## 我的理解
RPC 的价值不是“把 HTTP 换成另一个框架”，而是为服务间方法调用提供一套完整抽象。它让开发者更容易写分布式服务，但也容易让人忽略远程调用和本地调用的本质差异：远程调用会失败、会超时、会重复、会变慢。
