# 一致性模型

## 一句话解释
一致性模型定义的是：在多副本、多客户端、并发读写的系统里，**读操作可以看到什么样的写入结果**。它回答的不是“数据最终是否一样”，而是“写入之后，谁能在什么条件下、以什么顺序看见它”。

## 它回答的核心问题
- 它试图解释：为什么同样是“有多个副本”，不同系统的读写体验差异会很大。
- 它试图约束：写入成功后，其他客户端到底能看到多新、顺序能乱到什么程度。
- 它试图区分：线性一致、顺序一致、因果一致、最终一致之间的语义边界。

## 定义

```text
一致性模型 = 对并发读写在系统中的可见性语义的定义。
```

重点不在“副本最后会不会一样”，而在“中间过程允许读到什么”。

## 模型与假设
- 系统模型：多个副本、多个客户端、并发读写。
- 时间 / 顺序假设：不同模型对“真实时间”“程序顺序”“因果顺序”的依赖不同。
- 故障模型：通常默认网络延迟、分区、异步复制等现象可能存在。
- 观察视角：从客户端角度讨论读能否看到某次写，而不是单纯从存储内部视角讨论复制是否完成。

## 核心性质 / 关键结论
- 一致性模型是一个**强度谱系**，不是“强一致 vs 最终一致”二元对立。
- 语义越强，通常需要更多同步、协调与等待，成本越高。
- 语义越弱，系统越能换取可用性、延迟和吞吐，但业务层要承担更多不确定性。

## 为什么成立
因为在多副本环境里，“写完成了”和“所有读都立刻看到它”不是同一件事。不同系统通过不同同步机制、读路径和冲突解决策略，对外给出不同语义承诺，于是形成了一致性模型谱系。

## 成立边界
- 成立条件：必须先明确你讨论的是哪种复制模型、读写路径和故障条件。
- 失效条件：脱离系统模型谈一致性是无效的，比如不说明读是否走 leader，就无法判断读语义。
- 最容易忽略的前提：很多“强一致”的口语说法其实没有说清楚到底是线性一致、顺序一致，还是只是单会话不乱。

## 一致性谱系（重点复盘表）

| 模型 | 它真正保证什么 | 复盘时最该记住的重点 |
|---|---|---|
| [[线性一致性]] | 操作像在真实时间上的某个瞬间生效 | 写成功后，后续读不能读旧；代价高，常偏 CP |
| [[顺序一致性]] | 所有人看到同一全局顺序，但不要求尊重真实时间 | 比线性一致弱；“顺序自洽”不等于“实时可见” |
| [[因果一致性]] | 有因果关系的操作必须按因果顺序可见 | 重点是“先发帖再评论”这种依赖不能反过来 |
| [[最终一致性]] | 没有新写入时副本最终收敛 | 不承诺中间过程读到什么，重点在收敛与冲突处理 |

## 会话级补充保证
这些不是完整全局一致性模型，更像是“用户体验增强层”：

- **读己之写一致性**：自己写完，自己后续读取不应读旧。
- **单调读一致性**：同一会话内读取不能越读越旧。
- **单调写一致性**：同一会话的写要按程序顺序生效。
- **会话一致性**：上述保证的组合。

复盘重点：这些保证通常用于“最终一致系统里改善用户体验”，而不是替代全局强一致。

## 反例 / 边界案例
最典型的混淆是：

```text
系统 A：所有人看到同一顺序，但某个后完成的读仍可能读旧
系统 B：写一旦成功，后续读绝不读旧
```

很多人会把这两者都叫“强一致”，但严格说前者更接近顺序一致，后者才是线性一致。

## 与相邻概念的区别
- [[数据一致性]]：更偏“副本状态是否一样”的广义口语；一致性模型更偏“读写可见性语义”。
- [[CAP]]：CAP 讨论分区时 C/A 取舍；一致性模型讨论“你选的 C 到底是什么语义”。

## 与工程实践的关系
- 对设计的影响：先按数据类型分层，而不是系统全局只选一种一致性。
- 对实现的影响：强语义常依赖 leader、quorum、共识；弱语义要补会话保证、对账、冲突解决。
- 对排障的影响：先问“这条链路理论上承诺了什么”，再判断线上现象是不是 bug。

## 最小例子

```text
余额 / 锁 / 配额      -> 更接近线性一致
评论列表 / 时间线     -> 常接受最终一致或会话一致
先发帖再评论的场景    -> 因果一致比最终一致更贴切
```

## 常见误解
- “一致性只有强和弱两档”：不是，中间有很多层次。
- “最终一致就是不保证一致”：不是，它保证最终收敛，只是不保证实时可见性。
- “强一致一定最好”：错误。很多业务不值得为最强语义付出最高成本。

## 我的理解
一致性模型最值得记住的，不是定义本身，而是“你到底在向调用方承诺什么”。一旦承诺错了，后面再怎么优化复制和缓存，都只是让错误更难被发现。
