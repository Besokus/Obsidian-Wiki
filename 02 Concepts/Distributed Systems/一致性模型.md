# 一致性模型

## 一句话解释
一致性模型定义的是：在有多个副本、多个客户端、并发读写的分布式系统中，读操作应该看到什么样的写入结果。它回答的不是"数据最终是不是一样"，而是"写入发生后，其他客户端什么时候、以什么顺序、在什么条件下能读到它"。

## 一致性谱系

一致性模型是一个强度谱系：语义越强，成本越高（延迟、可用性、实现复杂度）。

| 模型        | 核心承诺                     | 典型成本        |
| --------- | ------------------------ | ----------- |
| [[线性一致性]] | 操作在真实时间上瞬间生效，写入成功后所有读取可见 | 延迟高，分区时需 CP |
| [[顺序一致性]] | 所有客户端看到同一全局顺序，但不依赖真实时间   | 仍需要全局协调     |
| [[因果一致性]] | 有因果关系的操作按因果顺序可见，无关系操作可并发 | 需要追踪依赖      |
| [[最终一致性]] | 无新写入时所有副本最终收敛，中间状态无保证    | 需要处理冲突与旧读   |

### 会话级补充保证
这些轻量级保证不定义全局语义，只确保同一会话内体验不乱，常与最终一致配合使用：

- **读己之写一致性**：客户端写入后，自己后续读取不读到旧值。实现：会话粘滞、读写走主。
- **单调读一致性**：同一客户端多次读取不会越读越旧。实现：会话粘滞、携带已见版本。
- **单调写一致性**：同一客户端的写入按程序顺序生效。实现：会话路由到同一节点。
- **会话一致性**：上述几种的组合，保证同会话内的一致性体验。

## 与 CAP / PACELC / BASE 的关系
- **CAP**：网络分区时，系统要在一致性与可用性之间选择。一致性模型定义的是"选择一致性时，承诺什么样的读写可见性"。
- **PACELC**：即使没有分区，系统也要在延迟与一致性之间权衡。
- **BASE**：更偏工程实践中"接受短暂不一致，通过机制收敛"的思路，通常与最终一致配合。

## 工程选型框架
设计系统时可以问：

```text
1. 这类数据读旧会造成什么后果？
2. 用户能否接受刚写完看不到？
3. 不同用户是否必须看到同一个顺序？
4. 是否存在因果依赖（先发帖再评论）？
5. 是否能通过异步修复收敛？
6. 冲突是否可以自动合并？
7. 网络分区时，是拒绝请求还是继续响应？
```

对应选择：

```text
不能错 → 线性一致 / 共识 / quorum
不能体验乱 → 会话一致 / 读己之写 / 单调读
有因果关系 → 因果一致
能接受延迟同步 → 最终一致
```

## 常见误区

### 误区 1：一致性只有"强一致"和"最终一致"
不对。一致性模型是一个谱系，中间还有顺序一致、因果一致、会话一致等多种模型。

### 误区 2：最终一致就是不保证一致
不对。最终一致要求最终收敛，有明确的同步、重试和冲突解决机制。

### 误区 3：强一致一定最好
不一定。对点赞数、搜索索引强行使用强一致，通常会浪费性能和可用性。

### 误区 4：一致性与可用性只能二选一
现实中，同一个系统可能对不同操作提供不同一致性：余额用强一致、订单列表用会话一致、搜索索引用最终一致。

## 我的理解
一致性模型不是"选哪个最好"的问题，而是"这个数据在什么故障条件下，允许读到多旧、允许顺序多乱"的问题。工程上不需要对所有数据用同一个承诺，重点是识别每条数据路径的一致性需求边界。
