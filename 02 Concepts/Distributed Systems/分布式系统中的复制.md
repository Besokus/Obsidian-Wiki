# 分布式系统中的复制

## 一句话解释
复制是把同一份数据维护在多个节点上，用来提升可用性、容错能力、读吞吐和跨地域访问性能。它的难点不在于"拷贝数据"，而在于网络延迟、节点故障和并发写入下如何维持可接受的一致性。

## 它解决什么问题
单节点架构有四个硬伤：单点故障导致整体不可用、读写能力受限于单机、磁盘损坏可能丢数据、跨地域用户访问延迟高。

复制把单点变成多点，但也引入了副本一致性问题。

- 可用性：一个节点宕机，其他节点可以继续服务。
- 容错性：单副本损坏时，可从其他副本恢复。
- 读扩展：读多写少时，让多个副本共同承担读请求。
- 延迟：跨地域部署副本，用户就近访问。

## 基本组成
| 组件 | 作用 |
|---|---|
| 副本（Replica） | 保存数据拷贝的节点，逻辑上同一份，物理上不同机器 |
| 主副本（Leader/Primary） | 接收写请求并同步给其他副本，某一时刻的数据权威来源 |
| 从副本（Follower/Secondary） | 接收 Leader 的更新并在本地重放，可承担读请求或故障时升主 |
| 复制日志（Replication Log） | 记录变更序列，从副本按顺序重放以追上 Leader 状态 |

## 三种复制模型
- [[主从复制]]：一个 Leader 接收写，同步给多个 Follower。最常见的模式。
- [[多主复制]]：多个节点都可接收写，互相同步。适合跨数据中心，但需处理写冲突。
- [[无主复制]]：没有固定 Leader，客户端向多个副本读写，依靠 quorum 保证一致性。

## 同步策略
不管哪种模型，都要回答"写入什么时候算成功"：

- **同步复制**：多个副本确认后才返回。数据安全，写延迟高。
- **异步复制**：Leader 本地成功后立即返回。写延迟低，Leader 宕机可能丢数据。
- **半同步复制**：Leader 本地 + 至少一个 Follower 确认后返回。折中方案。

## 复制 vs 备份
|         | 复制           | 备份             |
| ------- | ------------ | -------------- |
| 目的      | 高可用、故障切换、读扩展 | 历史恢复、误删恢复、灾难恢复 |
| 是否同步误操作 | 会            | 不应立即覆盖历史       |

误删数据后，复制会把删除同步到所有副本。所以复制不能替代备份。

## 工程判断
- 判断信号：需要多高可用性？能接受多大复制延迟？读是否必须实时？故障恢复时间目标（RTO）和数据丢失容忍度（RPO）是多少？
- 关键权衡：同步安全 vs 异步性能；单主简单 vs 多主低延迟 vs 无主高可用。
- 常见替代方案：缓存（不持久化）、定期快照 + WAL（非实时）、单机垂直扩展（不解决高可用）。

## 常见误解
- "有复制就不会丢数据"：异步复制下 Leader 返回成功后宕机，未同步数据可能丢失。
- "读从库一定安全"：从库可能落后，对余额、订单状态等场景危险。
- "复制可以替代备份"：复制会同步误删和误更新，不能替代备份。

## 我的理解
复制的核心不是"多存几份"，而是"在故障和延迟不可避免的前提下，让多个副本以可接受的方式保持一致"。选择复制方案本质上是在一致性、延迟、可用性和实现复杂度之间做交易。
