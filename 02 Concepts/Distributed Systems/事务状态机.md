# 事务状态机（Transaction State Machine）

## 一句话解释
分布式事务要可恢复、可重试、可复盘，必须把业务流程建模成**显式状态机**：每一步推进都落库，失败可重试，超时可补偿，最终能对账闭环。

---

# 1. 为什么一定要有状态机
在部分失败/不确定场景下，你需要回答：
```text
这笔业务现在进行到哪一步？
该继续推进还是补偿？
是否已经对外返回成功？
```
没有状态机就会变成“靠日志猜”，无法可靠恢复。

---

# 2. 一个可工作的最小形态
建议至少包含：
- `gid`（全局事务 id）
- `state`（当前状态）
- `step` / `cursor`（执行到第几步）
- `updated_at`（超时与巡检依据）
- `attempt`（重试次数）
- `last_error`（可观测）

状态推进建议是“单调 + 校验”：
```text
只允许从合法前置状态迁移到下一状态
重复请求直接返回当前状态（幂等）
```

---

# 3. 必须配套的能力
- **重试策略**：指数退避 + 抖动 + 最大次数 + 死信
- **幂等**：所有 step、补偿都必须幂等（见 [[幂等性]]）
- **超时与巡检**：卡住的事务要能被扫出来（pending/unknown）
- **人工介入入口**：极端长尾一定需要人工兜底（挂单/挂账/人工确认）
- **对账闭环**：最终一致系统必须能发现并修复不一致（见 [[对账]]）
