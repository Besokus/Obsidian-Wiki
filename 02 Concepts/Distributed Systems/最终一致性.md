# 最终一致性

## 一句话解释
最终一致性（Eventual Consistency）保证在没有新写入的情况下，所有副本最终会收敛到同一状态。它不承诺中间过程读到什么，只承诺停止写入后系统会同步一致。

## 它解决什么问题
强一致在高延迟、网络分区或高可用要求下代价过高。最终一致性用一种更松散、更高性能的方式，换取可用性和延迟。

- 可用性：部分副本不可用时，写入仍可成功。
- 延迟：本地写入即可返回，不需要跨副本确认。
- 正确性：需要应用层处理"读到旧值"和"写入冲突"。

## 什么时候用
- 读取不一致不直接影响业务正确性：点赞数、浏览量、热度排名。
- 数据最终通过异步渠道收敛：搜索索引、推荐特征、数据仓库同步。
- 有明确的冲突解决或对账机制的写操作。

## 什么时候不用
- 余额、权限、锁、配置等"一次性正确"的判断。
- 用户写完必须立刻看到自己写入的交互路径（除非配合读己之写）。
- 数据聚合适配或对账逻辑无法覆盖的场景。

## 核心机制
```text
1. 写入到达任意副本（取决于复制模型）。
2. 副本之间异步同步变更。
3. 没有新写入时，所有副本最终执行同一序列的变更。
4. 收敛依赖冲突解决策略：LWW、CRDT、向量时钟。
```

最终一致性不保证"多久会一致"——它只保证"最终"。

## 开发者视角
- 最终一致下读到旧值是常态，不是异常。
- 同一个客户端连续读可能先新后旧（非单调读），需要会话级补充保证。
- 写冲突需要显式解决：LWW 最简单但可能丢更新，CRDT 安全但有数据结构限制。
- 最终一致不意味着"不做任何事"——它需要 read repair、反熵、对账等机制确保收敛。
- 监控重点是：副本延迟（replication lag）、冲突率、读修复率。

## 工程判断
- 判断信号：业务能否接受"1秒/10秒/1分钟的短暂不一致"？是否有补偿或对账机制？
- 关键权衡：可用性和性能 vs 一致性和开发复杂度。
- 常见替代方案：读己之写（用户体验层面补偿）、因果一致（更精确的语义承诺）、带版本的对账。

## 最小例子
```text
T1: Replica A has x=1, Replica B has x=0
T2: Replica A has x=1, Replica B has x=0
T3: Replica A has x=1, Replica B has x=1  ← 最终收敛
```

## 常见误解
- "最终一致就是不保证一致"：它保证最终收敛，只是不保证何时。
- "最终一致最便宜"：实现好最终一致（带版本、冲突解决、反熵）的复杂度可能超过你的预期。
- "最终一致适合所有非核心数据"：如果用户体验需要"写完就看到"，即使业务不要求强一致，也要考虑会话级补充。

## 易混概念
- [[线性一致性]]：实时可见，代价高；最终一致：延迟同步，代价低。
- [[因果一致性]]：比最终一致更强，保证因果顺序的可见性。
- [[一致性模型]]：一致性模型的总览谱系。

## 我的理解
最终一致性不是"退而求其次"，而是一个经过权衡后的工程选择。它在高可用和大规模场景下几乎是必然的，但代价是业务层要为读到旧值和处理冲突做好准备。关键不是"能不能用最终一致"，而是"用最终一致时，不一致窗口的影响在不在业务可接受范围内"。
