# CAP

## 一句话理解
CAP 定理讨论的是：**一旦发生网络分区（Partition）**，分布式系统无法同时保证强一致性（Consistency，通常指线性一致性）和可用性（Availability，非故障节点必须返回非错误响应），只能在两者之间做取舍。

更准确地说：CAP 不是“系统永远只能选两个”，而是“**在 P 发生时必须在 C 和 A 之间选**”。

## 三个字母分别是什么
### C：Consistency（一致性）
CAP 语境下的 C 通常指 **线性一致性（Linearizability）**：一旦某个写入成功，后续读必须读到该写入结果，不能读到旧值。

注意：这里的 C 不是数据库 ACID 的 Consistency（约束一致性，例如外键、唯一性）。

### A：Availability（可用性）
CAP 的 A 指：**每个非故障节点收到请求后，都必须在有限时间内返回一个非错误响应**（不能因为联系不上其他节点就拒绝服务/无限等待）。

注意：这里的 A 不是工程口径的 “99.99% uptime”，而是更严格的语义定义。

### P：Partition Tolerance（分区容忍性）
P 指：系统必须能在网络分区、丢包、乱序、长延迟等“网络不可靠”情况下继续运行（用某种方式对外提供服务或拒绝服务），而不是假设节点之间永远可靠互通。

工程上只要是跨节点系统，网络问题无法完全避免，因此通常不能说“不要 P”。

## 发生分区时：CP vs AP
用两个副本 A/B 举例（都保存 `x`）：

```text
初始：A.x = 0, B.x = 0
分区：A <x---X---> B
写入：Client -> A 写 x=1 成功（此时 A.x=1, B.x=0）
```

此时另一个客户端去 B 读 `x`，系统就要选：

| 选择 | 分区时 B 的行为 | 结果 | 典型适用 |
|---|---|---|---|
| CP（偏一致性） | 拒绝/超时/返回错误，等待与 A 同步或达成一致 | 保 C，牺牲 A | 配置中心、分布式锁、元数据/协调系统、金融关键账务等 |
| AP（偏可用性） | 继续响应（可能返回旧值） | 保 A，牺牲 C | 更强调可用性与吞吐、可接受短暂不一致的业务读写场景 |

这里的 “CP/AP” 是**在发生网络分区时的偏好**，不是给某个数据库/中间件贴的永久标签。

## 为什么不能同时保证 C 和 A
分区发生时，节点之间无法可靠通信：

- 若要保证 C：必须阻止可能读到旧数据的请求（拒绝/等待），从而牺牲 A。
- 若要保证 A：必须继续响应请求，但因为无法确认是否有新写入，可能返回旧数据，从而牺牲 C。

## CA 系统存在吗
在“没有分区”的前提下（例如单机系统，或假设网络永远可靠且延迟可忽略），看起来可以同时满足 C 与 A。

但只要是跨节点系统，**P 是现实中必须考虑的故障模型**；CAP 的讨论点正是 “P 发生时怎么办”。

## CAP 与复制/一致性协议的关系
- 复制（主从、多主、无主）引入了写入传播延迟与副本分歧的可能性，使得 CAP 权衡变成现实工程问题。
- 共识系统（如 Raft/Paxos 的落地实现）通常更偏 CP：在分区时宁可不可用，也要避免对外暴露不一致的“权威状态”。

## CAP 与 PACELC
CAP 关注“发生分区时”的取舍；PACELC 补充了“**不发生分区时**也仍然有延迟与一致性的权衡”：

- P 发生时：在 A 与 C 之间取舍。
- Else（无分区时）：在 L（Latency）与 C 之间取舍。

## 常见误解
- “CAP 是简单三选二”：更准确是“发生分区时 C 与 A 二选一”。
- “P 可以不要”：跨节点系统无法回避网络不可靠，最多是降低分区概率或降低影响面。
- “CP 系统一定不可用”：不是“永远不可用”，而是在分区/无法达成一致时会拒绝部分请求以保持一致性语义。
- “AP 系统一定数据乱”：AP 允许短暂不一致，但很多系统会通过收敛机制、冲突解决、业务幂等/补偿等手段实现最终一致。
- “数据库/中间件可以直接贴 CP 或 AP 标签”：同一系统的不同操作/路径（读、写、强读、弱读、事务、配置开关）可能做不同权衡。

## 工程判断：如何选
### 什么时候偏 CP
- 读到旧数据/双写冲突的业务成本很高（财务、权限、配额、锁、配置发布等）。
- 宁可失败也不能错（Fail-fast / Fail-closed）。

### 什么时候偏 AP
- 更重视“能用”，可容忍短暂不一致（内容流、计数类、推荐、日志/埋点等）。
- 有清晰的收敛与补偿方案（最终一致、幂等、重试、对账、冲突解决等）。

## 一个简单判断框架
把问题具体化成一句话更好做决策：

> 网络分区时，这个请求是“宁可拒绝/超时”，还是“宁可返回可能过期的数据/接受冲突”，以及后续怎么收敛？
