# 故障模型

## 一句话解释
故障模型定义“系统会以什么方式出错”。它不是故障清单，而是系统设计的边界条件：你假设什么故障，就决定了协议需要多复杂、代价有多高、哪些正确性承诺才成立。

## 它回答的核心问题
- 它试图解释：为什么同样是“失败”，崩溃、超时、丢消息、错误响应、恶意作恶的处理成本完全不同。
- 它试图约束：系统、算法和一致性协议到底要对哪些失败负责。
- 它试图区分：哪些问题可以靠超时/重试解决，哪些已经超出了当前协议能力。

## 定义

```text
故障模型 = 对系统允许出现的错误行为的抽象假设集合。
```

它回答的不是“现实里会不会发生”，而是“设计时我们准备为哪些失败负责”。

## 模型与假设
- 系统模型：多个节点通过不可靠网络通信，节点和网络都可能出错。
- 时间 / 顺序假设：通常无法从超时可靠推出“对方一定挂了”。
- 故障模型：从 crash、omission、timing 到 response、Byzantine，强度逐级增加。
- 观察视角：从调用方看，失败可能表现为“不响应”“响应太慢”“响应错误”“行为不可信”。

## 核心性质 / 关键结论
- 故障模型越强，协议成本越高，延迟和复杂度也越高。
- 很多经典共识协议（如 Raft/Paxos）主要针对 **crash / crash-recover**，不是 Byzantine。
- 超时只能说明“不确定”，不能说明“对方一定失败”。
- 把不同故障类型混为一谈，会直接导致错误的恢复、重试或降级策略。

## 为什么成立
- 崩溃是“停止输出”；错误响应是“继续输出但结果不可信”，两者处理方式完全不同。
- 网络延迟、GC pause、队列堆积、磁盘抖动都会在调用方视角表现成“超时”，因此超时天然带不确定性。
- 一旦把容错目标从 crash 提高到 Byzantine，你必须增加投票、签名、仲裁、校验等额外机制，所以成本陡增。

## 成立边界
- 成立条件：先明确系统真的主要面对什么故障，再选择协议。
- 失效条件：如果你把“可能错误响应/恶意行为”的系统，按 crash-only 去设计，就会在错误数据传播时失守。
- 最容易忽略的前提：`crash-stop` 和 `crash-recover` 不是一回事；恢复后的节点可能带着过期状态回来。

## 反例 / 边界案例
最典型的误判是：

```text
客户端超时
=> 误以为请求失败
=> 直接重试
=> 原请求其实已经成功
=> 产生重复扣款 / 重复下单
```

这里真正的故障模型不是“失败”，而是“遗漏/时序导致的状态不确定”。

## 与相邻概念的区别
- [[网络分区]]：网络分区是某一种具体故障现象；故障模型是对所有允许故障类型的抽象边界。
- [[部分失效]]：部分失效强调“不是整个系统一起坏”；故障模型强调“坏的方式是什么”。

## 与工程实践的关系
- 对设计的影响：决定你该用 crash 容错协议、幂等重试，还是需要更强的审计、签名、BFT。
- 对实现的影响：直接影响超时、重试、熔断、降级、对账、恢复日志的设计。
- 对排障的影响：先分清是“无响应”“慢”“错结果”“不可信”，再决定排障路径。

## 最小例子

```text
场景 A：节点宕机不再响应         => crash
场景 B：请求已执行但响应丢失     => omission / uncertainty
场景 C：节点还在返回数据，但数据错 => response failure
场景 D：节点给不同人返回互相矛盾结果 => Byzantine
```

## 常见误解
- “超时就是失败”：不是，超时只说明你现在不知道结果。
- “拜占庭就是网络分区”：不是，分区是可达性问题；拜占庭是行为可信性问题。
- “企业内部服务不用考虑错误响应”：很多线上事故恰恰来自“服务没挂，但返回了错结果”。

## 我的理解
故障模型是所有分布式设计的起点，不是附录。你不先说明“准备承受什么样的失败”，后面关于一致性、共识、重试、事务的讨论都会变成漂浮在空中的结论。
