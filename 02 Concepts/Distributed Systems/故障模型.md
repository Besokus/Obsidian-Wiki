# 故障模型

## 一句话解释
故障模型定义系统允许出现哪些错误行为，以及协议要为哪些错误负责。它是分布式系统所有正确性讨论的起点。

## 它回答的核心问题
- 它试图解释：系统到底会以什么方式出错。
- 它试图约束：协议需要容忍到什么程度的错误。
- 它试图区分：崩溃、漏消息、超时、错误响应和恶意行为不是同一类问题。

## 它解决什么问题
- 为共识、复制、重试、降级等协议提供边界。
- 避免把不同类型的错误混为一谈。
- 决定系统应该采用多强的容错机制。

## 重点部分详解
- crash-stop / crash-recover：节点停止服务，或者恢复后可能带着旧状态回来。
- omission：消息丢失、请求丢失、响应丢失都属于不完整通信。
- timing：消息到达太慢，导致超时和不确定性。
- response failure：节点返回了错误结果。
- Byzantine：节点可能任意撒谎、伪造、前后不一致。
- 故障模型越强，协议成本越高；大多数工程系统主要假设 crash-recover。

## 核心机制
```text
1. 先明确系统允许哪些错误。
2. 再决定协议要提供多强的容错。
3. 不同故障类型使用不同恢复和检测手段。
4. 不把超时、错误响应、恶意行为混为一谈。
```

## 典型故障层级
| 故障类型 | 含义 | 常见处理 |
|---|---|---|
| Crash | 节点停止工作 | 超时、重试、选主 |
| Omission | 请求/响应/消息丢失 | 超时、重发 |
| Timing | 太慢或延迟过高 | 超时、退避 |
| Response | 返回错误结果 | 校验、幂等、对账 |
| Byzantine | 任意恶意行为 | 签名、投票、仲裁 |

## 易混概念
- [[超时]]：只能表示“不确定”，不能表示“对方一定失败”。
- [[网络分区]]：是一类具体故障现象，不等于整个故障模型。
- [[部分失效]]：说明系统不一定整体挂掉。

## 我的理解
故障模型决定了你能用什么协议，也决定了你不能做什么假设。很多设计错误，本质上都是把一个更强的故障模型，误当成了一个更弱的故障模型。
