# 共识机制（入门整理）

## 一句话解释
共识机制解决的是：在节点可能宕机、网络可能延迟/丢包/乱序、甚至出现网络分区的情况下，多个副本如何对同一条“决定”（写入、配置变更、选主结果、日志顺序）达成一致，并且不产生互相矛盾的结果。

> 核心目标不是“更快”，而是“故障下不做错”（Safety 优先）。

## 0. 共识在系统里通常负责什么
共识常见落点是“控制面/元数据”，例如：

```text
谁是 leader？
某条日志是否 committed？
某个配置版本是否生效？
某个分布式锁是否被授予？
某个节点是否还能提供写服务？
```

业务数据面（高吞吐、可异步修复的数据）通常不直接用共识承担每条写入（见“工程适用性”）。

---

# 1. 先统一：故障模型（你面对的不是“正常网络”）
共识协议（如 Raft/Paxos）通常针对 **Crash Failure**：节点不会恶意撒谎，但会宕机/变慢/重启/丢消息。

你需要默认以下都可能发生：

- **节点宕机**：突然不响应。
- **网络延迟**：收不到响应 ≠ 对方宕机。
- **网络分区**：集群被切开，分区之间互相不可达。
- **消息丢失/重复/乱序**：必须用 `term/index/message id` 等标识去判定新旧与有效性。

> 新手易错点：把“收不到响应”当成“节点死了”，然后做出不安全决策。

---

# 2. 共识要保证什么：Safety & Liveness（优先级不同）

## 2.1 Safety（宁可停，也不能错）
Safety 要求系统不能做出矛盾决定，例如：

```text
Node A 认为 log[10] = set x = 1 已 committed
Node B 认为 log[10] = set x = 2 已 committed
```

出现这种情况基本就不可恢复（相当于“历史被篡改”）。

## 2.2 Liveness（最终能继续）
在网络恢复、故障不超过容忍范围时，系统应当最终恢复处理能力（能选出新 leader，能继续 commit）。

> 现实里通常：Safety > Liveness。分区/不确定时可以拒绝服务，但不能提交错误结果。

---

# 3. Quorum（多数派）是共识的支点
对于 `2f+1` 个节点，多数派是 `f+1`。关键性质是：

```text
任意两个多数派一定有交集
```

这保证了：只要一条决定被多数派接受，未来任何能形成多数派的一侧，都会“继承”这条决定的证据（交集节点携带历史）。

## 3.1 为什么多数派能防脑裂
网络分区时，只有**能形成多数派**的一侧可以 commit；少数派即使“自认为 leader”，也无法满足提交条件，因此无法对外产生“已提交”的新历史。

## 3.2 节点数与容错

|节点数|多数派|可容忍宕机|
|--:|--:|--:|
|1|1|0|
|3|2|1|
|5|3|2|
|7|4|3|

工程最常见 3/5 节点；2 节点在“任一节点故障时无法形成多数派”，因此通常不选。

---

# 4. Leader：把“多方竞争”降维成“单点排序 + 多副本复制”
多数工程化共识会选出 leader，形成典型写路径：

```text
Client -> Leader -> Followers
```

Leader 做两件关键事：

1) **排序**：把并发写入变成一个全序日志（log）。
2) **推进提交**：把日志复制到多数派，满足 commit 条件后再对外宣布成功。

> 重要：Leader 不是“绝对权威”。`leader append != committed`。

---

# 5. 状态机复制：共识通常在“决定一串操作的顺序”
工程里共识更常见的形式不是“决定一个值”，而是“决定一串命令的顺序”，即 **State Machine Replication**：

```text
相同初始状态 + 相同操作序列 + 相同执行顺序 => 相同最终状态
```

日志项（以 Raft 为例）通常包含：

```text
index: 日志位置
term: 产生该日志的任期
command: 要执行的状态变更
```

`index + term` 是判断日志匹配/冲突/新旧的重要依据。

---

# 6. 四个关键动词：append / replicate / commit / apply
这是入门最容易混淆的部分。建议牢牢记住链路：

```text
append -> replicate -> commit -> apply
```

- **append**：leader（或 follower）把日志追加到本地存储。
- **replicate**：leader 把日志发送给 followers，使其也 append。
- **commit**：leader 观察到该日志已在多数派持久化 → 该日志不可被合法覆盖。
- **apply**：节点把 committed 日志对应的命令执行到状态机（真正改变对外可见的状态）。

> 关键：对客户端“成功返回”通常绑定在 commit（以及正确的 apply/读路径）上，而不是 append。

---

# 7. 选主与 term：让“旧 leader”自动退位
Raft 的简化选举流程：

```text
Follower election timeout
-> Candidate
-> term + 1
-> RequestVote
-> 获得多数票 => Leader
```

`term` 是逻辑任期（逻辑时间），用于：

- 区分新旧 leader（更大 term 更新）。
- 识别过期消息（旧 term 的消息应丢弃/拒绝）。
- 防止旧 leader 在恢复后继续提交（看到更大 term 就必须 step down）。

随机化 election timeout 用于减少多 candidate 同时竞争导致的反复选举。

---

# 8. 脑裂与恢复：为什么“旧 leader 恢复后不能继续提交”

## 8.1 脑裂是什么
脑裂是：多个节点同时认为自己是 leader，并都对外接受写入且“提交成功”。如果两边都能 commit，就会产生无法合并的分叉历史。

共识通过“多数派提交 + 多数派交集”避免它：

```text
leader 必须获得多数票
日志必须多数派确认才能 committed
任意两多数派必有交集 => 不会出现两个都能合法 commit 的 leader
```

## 8.2 旧 leader 恢复后会发生什么
典型场景：旧 leader 在网络分区期间可能 append 了未提交日志，但它联系不到多数派，因此无法 commit。

当网络恢复、它发现更大的 term 时，会：

```text
step down -> follower
```

从而停止以“leader 身份”推进提交。

---

# 9. 日志冲突与修复：为什么“未提交可覆盖，已提交不可覆盖”
Leader 切换常导致日志不一致。规则非常硬：

```text
已 committed 的日志不能被覆盖
未 committed 的日志可以被新 leader 覆盖
```

直觉解释：只有多数派确认过的历史才是“系统共同承认”的历史；否则就是“某个节点私自记的草稿”，当它和新的多数派历史冲突时必须被回滚/覆盖。

---

# 10. 与 CAP 的关系：共识通常偏 CP
网络分区时：

- 多数派一侧：还能形成 quorum → 可以继续 commit（系统可用）。
- 少数派一侧：无法形成 quorum → 不能 commit（拒绝写入/阻塞）。

因此共识系统通常偏 **CP**：优先一致性，在分区时牺牲少数派可用性。参见 [[CAP]]、[[PACELC]]。

---

# 11. 与线性一致性的关系：共识提供基础，但读路径也要配合
共识写路径常设计为：

```text
Client -> Leader -> replicate to majority -> commit -> apply -> reply
```

它为 [[线性一致性]] 提供了基础：写一旦成功返回，后续读在正确实现的前提下应该能观察到该写。

但“日志 committed”不自动意味着“任意读都线性一致”；读路径常见策略：

```text
读走 leader
leader 读前确认自己仍是 leader
quorum read / ReadIndex / lease read
```

---

# 12. 工程适用性：哪些数据应该用共识
共识成本高（延迟、吞吐、复杂度），适合：

```text
少量、关键、控制面、元数据、强正确性（宁可不可用也不能错）
```

不适合：

```text
高频日志、点赞/浏览计数、feed、推荐特征、大规模事件流、可异步修复数据
```

这些更适合 [[分布式系统中的复制]]、消息队列、最终一致性/BASE 等思路。

---

# 13. 常见误区速查
- **误区**：共识就是复制。  
  **纠正**：复制关心“传过去没有”；共识关心“系统是否共同承认成立（committed）”。
- **误区**：leader 写入成功就算成功。  
  **纠正**：`append != commit`，必须满足多数派提交条件。
- **误区**：网络分区两边都应该能写。  
  **纠正**：只能多数派一侧能 commit，否则脑裂。

---

# 14. 最小掌握标准：逐题回答

> 目标：每题能用 3~8 句话讲清楚，能画出关键流程（leader/多数派/term/commit）。

## 1) 共识和普通复制有什么区别？
普通复制（如主从复制）强调“数据从主传到从”；主返回成功时，并不必然意味着该写在故障下不会丢。共识把“返回成功”的语义绑定到“多数派持久化 + 合法提交（committed）”，强调系统层面共同承认一条操作成立。换句话说：复制解决“传播”，共识解决“确定性与不可矛盾的决定”。

## 2) 为什么共识需要多数派？
多数派的关键性质是“任意两个多数派一定有交集”。这让已经被多数派接受的决定，在未来任意能形成多数派的过程中都不会完全丢失（交集节点携带历史）。没有交集就可能出现两组互不相交的“合法集合”，从而产生不可合并的分叉历史（脑裂）。

## 3) 3 节点为什么只能容忍 1 个节点故障？
3 节点多数派是 2。只要再少一个节点就无法形成多数派，也就无法 commit 新日志（Safety 可能还能维持，但 Liveness 丢失）。因此最多容忍 1 个节点故障：剩下 2 个节点仍能形成多数派继续工作。

## 4) 什么是 leader？leader 为什么不能单方面提交？
leader 是当前任期内被多数派承认的协调者，用来排序写入并推进日志复制。它不能单方面提交，因为单节点决定在分区/宕机时会造成两个分区都“各自提交成功”的脑裂风险。提交必须依赖多数派确认，从而借助“多数派交集”保证历史唯一且可继承。

## 5) 什么是 term？它解决什么问题？
term 是逻辑任期（逻辑时间），把系统推进划分为一段段“谁是 leader”的时代。它用于区分新旧 leader、识别过期消息，并在旧 leader 恢复后通过“看到更大 term 必须退位”阻止其继续提交。term 让系统状态单调向前推进，减少“旧信息回流导致错误决定”。

## 6) append、replicate、commit、apply 有什么区别？
append 是写入本地日志；replicate 是 leader 把日志发送到其他节点并让它们也 append；commit 是 leader 观察到该日志已在多数派持久化，于是该日志成为系统承认的历史（不可被合法覆盖）；apply 是把 committed 日志执行到状态机，真正改变可见状态。新手要牢牢记住：`append != commit`，`commit != apply`。

## 7) leader 挂了之后如何选新 leader？
followers 通过 heartbeat 维持“我知道谁是 leader”。如果一段时间收不到 heartbeat，会触发 election timeout，节点转为 candidate、term+1 并向其他节点 RequestVote。拿到多数票的 candidate 成为新 leader。随机化 election timeout 的目的是降低多个 candidate 同时竞争造成的反复选举。

## 8) 什么是脑裂？共识如何避免脑裂？
脑裂是多个节点同时认为自己是 leader，并都对外接受写入且“提交成功”，导致出现两条不可合并的历史分叉。共识通过两条硬约束避免：leader 必须拿到多数票；日志必须多数派确认才能 committed。由于任意多数派都有交集，所以不可能出现两个互相独立且都能合法 commit 的 leader。

## 9) 旧 leader 恢复后为什么不能继续提交？
旧 leader 在分区期间可能还自认为 leader，但它无法联系多数派，所以无法满足 commit 条件。网络恢复后，它会从其他节点/请求中看到更大的 term，于是必须 step down 成 follower，承认自己过期并停止推进提交。否则就可能在“多数派已经换了 leader”的情况下继续对外宣布成功，破坏 safety。

## 10) 已提交日志为什么不能被覆盖？
已提交（committed）意味着该日志已经在多数派持久化；未来任何能形成多数派的 leader 选举都必须与这份多数派有交集，从而继承这条历史。如果允许覆盖 committed 日志，就等于允许系统对外承诺过的历史被改写，会造成不同节点对同一 index 的 committed 值不一致（灾难）。

## 11) 未提交日志为什么可以被覆盖？
未提交日志没有被多数派共同承认，本质上是“某个节点的草稿”。在 leader 切换后，新 leader 需要让全体日志收敛到一个统一历史；与新多数派历史冲突的未提交日志必须回滚/覆盖，否则会阻碍日志一致与后续提交。覆盖未提交日志不会破坏对外承诺，因为它从未被合法宣布为成功。

## 12) 为什么共识系统通常偏 CP？
分区时如果仍要保持一致性（C），就必须保证提交依赖多数派（quorum）。少数派无法形成 quorum，因此只能拒绝/阻塞写入（牺牲该分区可用性 A）。但多数派一侧仍可继续服务，所以不是“整个系统都不可用”，而是“只有能联系到多数派的一侧可用”。

## 13) 共识如何帮助实现线性一致性？
线性一致性要求操作看起来像在真实时间上“瞬间生效”。共识通过“多数派提交 + leader 顺序化日志 + 提交后再对外返回成功”让写入具有强确定性：一旦成功返回，就有多数派证据保证这条写不会丢。配合正确读路径（例如读走 leader 并确认 leader 身份，或 quorum read/ReadIndex），可实现对外可见的线性一致语义。参见 [[线性一致性]]。

## 14) 为什么不是所有数据都应该用共识？
共识每次写通常要跨节点通信、等待多数派并进行持久化，导致延迟更高、吞吐更低、实现更复杂。很多业务数据（计数、日志、事件流、推荐特征）容忍短暂不一致且可异步修复，用最终一致/异步复制更划算。共识应当集中用于“少量但关键”的控制面/元数据，否则就是把系统成本花在不需要的地方。
