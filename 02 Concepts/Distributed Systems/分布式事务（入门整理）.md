# 分布式事务（入门整理）

## 原子笔记索引（建议从这里跳转）
- 协议/模式：[[2PC]]、[[3PC]]、[[TCC]]、[[Saga]]、[[Outbox Pattern]]、[[Transactional Message]]
- 工程底座：[[幂等性]]、[[补偿]]、[[事务状态机]]、[[对账]]

## 一句话解释
分布式事务解决的是：**一次业务操作跨多个服务/数据库/分片/外部资源时，如何在“部分失败 + 超时不确定 + 重试重复”的现实里，让系统在业务语义上要么达成一致成功，要么能被一致地恢复/补偿到可解释状态。**

> 关键不是“永远回滚到原状”，而是“结果可解释、可恢复、可对账”。

---

# 1. 为什么难：本地 ACID 出了边界就不再自动成立

单机事务能靠数据库 ACID 兜底：

```text
BEGIN
  update A
  update B
COMMIT
```

分布式场景状态分散在不同边界（不同 DB/不同服务/不同资源）：

```text
Order DB
Inventory DB
Payment DB
Coupon DB
Message Queue
```

问题的本质是：**多个本地事务之间没有天然原子性**。当出现“中间某一步失败/超时/不确定”时，你无法用一个数据库回滚其他系统的状态。

---

# 2. 你真正要解决的四类问题（工程视角）

## 2.1 原子提交（Atomic Commit）
多参与者要么一起提交，要么一起不提交；避免出现“部分提交、部分回滚”的长期不一致。

## 2.2 部分失败（Partial Failure）与“不确定”
分布式世界最痛的不是“失败”，是**不知道对方到底成功没成功**：

```text
超时 ≠ 失败
可能是：请求丢了 / 响应丢了 / 对方慢 / 对方处理成功但回包超时
```

因此你必须设计：重试、幂等、状态记录、恢复流程。

## 2.3 并发冲突
例如超卖/重复扣款：并发写同一资源，需要锁/版本号/CAS/唯一约束/预留机制等并发控制。

## 2.4 故障恢复
系统宕机/重启/升级后，你必须能回答：

```text
这个事务现在处于什么状态？
下一步该继续、回滚、补偿，还是等待人工？
```

所以“日志 + 状态机 + 可重放/可修复”是分布式事务的刚需。

---

# 3. 核心建模：全局事务、参与者、状态机

## 3.1 角色
- **Coordinator（协调者）**：驱动流程（推进、超时、重试、补偿）。
- **Participant（参与者）**：持有资源的系统（订单/库存/支付/优惠券…），负责本地事务。

## 3.2 两层事务
- **Local Transaction（本地事务）**：每个服务内部仍靠 DB 事务保证本地一致。
- **Global Transaction（全局事务）**：跨服务的业务流程，需要显式状态与协调。

## 3.3 状态机（建议显式持久化）
状态命名因系统而异，但建议至少具备“可恢复语义”：

```text
INIT
TRYING / PREPARED
CONFIRMING / COMMITTING
CANCELLING / COMPENSATING
SUCCEEDED / FAILED / CANCELLED
```

> 有状态机，才谈得上恢复、排查、对账与告警。

---

# 4. 方案一：2PC（Two-Phase Commit）

2PC 解决的是“多参与者原子提交”。
详见：[[2PC]]

## 4.1 两个阶段（必须会画图）

### Phase 1：Prepare / Vote
协调者询问参与者能否提交；参与者执行“可提交准备”（常伴随锁资源）并返回 `YES/NO`。

### Phase 2：Commit / Rollback
若全员 `YES` → 协调者广播 `COMMIT`；否则广播 `ROLLBACK`。

## 4.2 2PC 的核心问题：阻塞（Blocking）
典型卡死场景：

```text
参与者都 YES 并锁住资源
协调者在发 COMMIT 前宕机 / 分区
参与者不知道最终决策 => 为了 safety 只能等待
```

因此 2PC 常见副作用：
- 协调者单点影响推进（除非你做了更复杂的 HA/日志复制）。
- 锁持有时间长（对吞吐/延迟很敏感）。
- 分区下可用性差（容易卡住）。

> 结论：2PC/XA 更适合“短事务 + 同构资源 + 能接受阻塞成本”的场景。

---

# 5. 方案二：3PC（Three-Phase Commit）——知道即可
3PC 试图通过增加中间阶段降低阻塞概率，但在现实异步网络模型下仍可能出现不安全/复杂性过高的问题，工程中普遍不如 2PC/TCC/Saga/消息方案常见。
详见：[[3PC]]

---

# 6. 方案三：TCC（Try / Confirm / Cancel）

TCC 的核心是：把“提交/回滚”从数据库层拉到业务层，实现**资源预留**。
详见：[[TCC]]

## 6.1 三个阶段
- **Try**：预留资源（冻结库存/冻结优惠券/冻结余额）。
- **Confirm**：把预留转为正式扣减（最终提交）。
- **Cancel**：释放预留（取消）。

## 6.2 TCC 的三个坑（必须掌握细节）

### 1) 幂等（Idempotency）
Confirm/Cancel 可能被重复调用（超时重试、恢复重放、消息重复），必须做到“多次执行结果一致”。

### 2) 空回滚（Empty Cancel）
Try 可能根本没成功（甚至请求未到），但 Cancel 已触发；Cancel 必须能识别“无资源可释放”的情况并安全返回成功。

### 3) 悬挂（Hanging / Late Try）
Try 超时触发 Cancel 后，Try 请求“后来才到”并成功冻结资源，导致系统进入错误状态。解决思路通常是：

```text
记录事务状态（已取消）并在 Try 时拒绝/幂等处理
或者使用业务 token/版本号防止 late Try 生效
```

> 结论：TCC 适合“库存/优惠券/余额”这类可预留资源的核心交易链路，但实现侵入性强、心智负担大。

---

# 7. 方案四：Saga（本地事务 + 补偿）

Saga 把长事务拆成一串本地事务 `T1..Tn`，失败时执行补偿 `Ck` 回滚业务效果（注意：不是 DB rollback）。
详见：[[Saga]]、[[补偿]]

## 7.1 两种实现方式
- **编排式（Orchestration）**：由一个 orchestrator 驱动下一步与补偿（更可控、易观察）。
- **协同式（Choreography）**：服务间通过事件协作推进（解耦但更难排查与治理）。

## 7.2 关键认知：补偿不是回滚
很多动作不可逆或代价高：

```text
短信无法撤回
已发货无法“回滚发货”
外部支付无法瞬时撤销
```

所以 Saga 的目标是“让业务回到可接受的补偿态”，而非“回到像没发生过一样”。

> 结论：Saga 适合微服务长流程（下单→发货→积分…），但不提供强隔离，需要设计中间态与用户体验（例如“订单处理中/待确认”）。

---

# 8. 方案五：本地事务 + MQ（为什么会出原子性坑）
最常见形态是“本地事务提交后通知下游”，但存在经典原子性问题：

```text
本地事务成功，但消息没发出去（消息丢）
消息发出去了，但本地事务回滚（脏消息）
```

因此需要 Outbox 或事务消息来把“写库”和“发消息”在语义上绑在一起。

---

# 9. 方案六：Transactional Outbox（强烈推荐掌握）
详见：[[Outbox Pattern]]

## 9.1 核心思路
在同一个本地事务中同时写业务表和 outbox 表：

```text
BEGIN
  update business tables
  insert outbox(message)
COMMIT
```

再由 relay/worker 异步把 outbox 投递到 MQ。

## 9.2 你必须接受的现实
- 消息**可能重复**：relay 可能重复投递 → 消费者必须幂等。
- 消息**可能延迟**：轮询、积压、限流导致延迟 → 业务要容忍并监控。
- outbox **需要清理/归档**：否则表会膨胀。

> Outbox 的价值：把“业务落库”和“产生消息”变成同一事务内的原子结果，从而大幅降低丢消息概率。

---

# 10. 方案七：事务消息（MQ 原生支持时的选择）
详见：[[Transactional Message]]
事务消息本质上也是解决“本地事务 + 消息发送”的原子性问题，只是把协议与回查机制放在 MQ/Broker 侧（例如 half message + 回查）。

工程对比（直觉）：
- Outbox：依赖 DB + 自建 relay；简单可靠，但有 polling/延迟与表治理成本。
- 事务消息：依赖 MQ 能力；减少自建，但更绑定 MQ 供应商语义与运维复杂度。

---

# 11. 工程基本功：比背协议更重要
不管你用 TCC/Saga/Outbox 还是事务消息，这些都是底座：

## 11.1 幂等（Idempotency）
详见：[[幂等性]]
常见实现：

```text
全局事务 ID（gid）+ 幂等表 / 去重表
唯一约束（唯一键/幂等 key）
状态机检查（只有某些状态允许推进）
版本号 / CAS
```

## 11.2 重试（Retry）
重试必须配合：

```text
幂等
退避（指数退避 + 抖动）
最大次数 / 死信队列
```

## 11.3 补偿（Compensation）
详见：[[补偿]]
补偿要设计“可接受的业务态”，并提供可观测性与人工介入入口。

## 11.4 状态机与审计
详见：[[事务状态机]]
事务状态、关键决策、请求/响应摘要、错误码必须能追踪；否则分布式事务无法复盘。

## 11.5 对账（Reconciliation）
详见：[[对账]]
最终一致系统的兜底：订单 vs 支付、订单 vs 库存、订单 vs 优惠券…定期对账发现异常并修复/补偿/人工处理。

---

# 12. 选型速查（按“你要的语义”）

## 12.1 强一致、短事务、同构资源（同一 DB/同类 DB）
- 候选：2PC/XA
- 风险：阻塞、锁时间长、分区可用性差

## 12.2 资源预留类（库存/优惠券/余额）
- 候选：TCC
- 风险：实现侵入性高；必须处理幂等/空回滚/悬挂

## 12.3 长流程微服务（多步骤、多下游、长耗时）
- 候选：Saga + 状态机 + 补偿
- 风险：隔离弱；要接受中间态与补偿失败

## 12.4 本地事务后通知下游（事件驱动、异步扩展）
- 候选：Outbox / 事务消息
- 风险：重复/延迟；消费者幂等与监控治理必做

## 12.5 高风险金融/支付
- 候选：通常是“状态机 + 强审计 + 对账 + 人工兜底”的组合拳，协议只是其中一部分。

---

# 13. 电商下单：把多条链路拆开（建议的心智模型）

## 13.1 核心交易链路（强语义）

```text
创建订单
锁/扣库存
冻结/核销优惠券
支付
```

候选：TCC / Saga（核心步骤）/ 本地事务 + 状态机（严格推进与恢复）。

## 13.2 事件通知链路（可最终一致）

```text
发送订单创建消息
更新积分
更新搜索索引
更新推荐特征
```

候选：Outbox + MQ + 消费者幂等（不要把这些硬塞进强事务）。

## 13.3 兜底链路（最后防线）

```text
订单 vs 支付 对账
订单 vs 库存 对账
订单 vs 优惠券 对账
```

发现异常：补偿 / 修复状态 / 人工处理。
