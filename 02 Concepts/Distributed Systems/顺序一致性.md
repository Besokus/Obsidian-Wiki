# 顺序一致性

## 一句话解释
顺序一致性要求所有客户端看到相同的全局操作顺序，且每个客户端自己的程序顺序在该全局顺序中保持不变；但它不要求该顺序与真实时间一致。

## 它解决什么问题
强一致（线性一致）依赖精确的时间同步，在分布式系统中代价高且实现复杂。顺序一致性放松了"必须按真实时间生效"的要求，让系统可以用更低的协调成本提供可预测的全局顺序。

- 正确性：所有客户端看到同一操作顺序，不会出现互相矛盾的观察。
- 可理解性：比线性一致弱，但比最终一致强——中间态仍然存在，但顺序不乱。
- 性能：不需要真实时间戳同步，延迟低于线性一致。

## 什么时候用
- 需要全局顺序但不强求实时可见性——即客户端可以读到稍旧的值，但顺序不能乱。
- 多端复制的场景中，只要保证所有客户端看到相同的写入序列。

## 什么时候不用
- 需要"一旦写入成功，后续读取必须立即看到"的语义（此时需要线性一致）。
- 业务完全不用关心全局顺序，最终一致就足够。

## 核心机制
```text
1. 操作被排列成一个全局序列。
2. 每个客户端自己的操作在该序列中保持其程序顺序。
3. 不同客户端的并发操作可以任意排列。
4. 所有客户端看到的全局序列相同。
```

## 重点性质详解
- 关键性质：所有人看到同一个全局顺序，但这个顺序不要求尊重真实时间。
- 为什么重要：它比最终一致更可预测，但比线性一致更便宜。
- 反例 / 边界：真实时间上先完成的写，在全局序列里仍可能排后。
- 和相邻概念的差别：线性一致要求真实时间也对齐，顺序一致不要求。

## 与线性一致性的区别
```
线性一致性：
  T1: Client A write x = 1 (real time 10:00:00.100)
  T2: Client B read x     (real time 10:00:00.200)
  → 如果 T2 在真实时间上发生在 T1 之后，必须读到 1。

顺序一致性：
  T1: Client A write x = 1
  T2: Client B read x → 可能读到 0
  → 只要系统解释为"全局序列里读发生在写之前"，即不违反顺序一致。
```

核心差别：线性一致依赖真实时间，顺序一致不依赖。

## 开发者视角
- 顺序一致性是理解更复杂模型的基础：在线性一致和最终一致之间的模型多在此之上做取舍。
- 顺序一致性允许"时间倒流"现象：真实时间上先发生的写入，在全局序列中可能排在后面。
- 很多数据库的"快照隔离"级别接近顺序一致性，但通常不严格等同。

## 易混概念
- [[线性一致性]]：顺序一致性 + 真实时间约束 = 线性一致性。
- [[因果一致性]]：只要求有因果关系的操作按因果顺序可见，比顺序一致更弱、更便宜。

## 我的理解
顺序一致性是一个"理论上很干净"的模型——它用"全局序列"的抽象描述了强一致的基本形态，又避开了真实时间同步的复杂度。工程上很少直接对外承诺"我们提供顺序一致性"，但很多系统的内部行为只有用这个模型才能准确描述。
