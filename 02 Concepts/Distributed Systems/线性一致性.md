# 线性一致性

## 一句话解释
线性一致性（Linearizability）要求每个操作都像是在 **真实时间轴上的某一个瞬间原子生效**。如果操作 A 在真实时间上先完成，操作 B 后开始，那么所有人都必须看到 A 在 B 之前发生的效果。

## 它回答的核心问题
- 它试图解释：多副本系统中，写入成功后，后续读取到底该不该立刻看到它。
- 它试图约束：系统对外表现必须像“只有一个正确副本”。
- 它试图区分：什么叫“实时生效”，什么只是在某个全局顺序上自洽。

## 定义

```text
线性一致性要求：
1. 每个操作看起来都在某个瞬间生效（linearization point）
2. 如果 A 在真实时间上先于 B 完成，则系统中的等价顺序也必须是 A 在 B 前
```

它关心的是 **单个操作的原子可见性**，不是多操作事务隔离。

## 模型与假设
- 系统模型：多个副本、多个客户端、并发读写。
- 时间 / 顺序假设：需要尊重“真实时间上的先后完成关系”。
- 故障模型：通常在 crash / 分区模型下通过 quorum 或共识实现。
- 观察视角：从客户端看，系统像单副本一样，没有“读到旧值但写已经成功”的现象。

## 核心性质 / 关键结论
- 写成功后，后续读取不能再看到旧值。
- 系统行为等价于一个单副本按某个顺序串行执行。
- 线性一致性比顺序一致性更强，因为它要求保留真实时间约束。
- 线性一致性通常意味着在分区时偏向 CP，而不是 AP。

## 为什么成立
- 只要每个操作都能找到一个合法的 linearization point，且这些点不违反真实时间顺序，系统就能被解释为“单副本在依次执行”。
- 一旦后完成的读还能看到旧值，就找不到一个同时满足“真实时间先后”和“单副本执行”的解释顺序，线性一致性就被破坏。

## 成立边界
- 成立条件：写入要得到足够强的同步确认；读取要确保读到的是最新已提交状态。
- 失效条件：异步复制后立即从落后副本读，通常不能保证线性一致。
- 最容易忽略的前提：写路径线性一致，不等于读路径天然线性一致。读策略同样必须设计正确。

## 反例 / 边界案例

```text
T1: Client A 写 x = 1，收到“成功”
T2: Client B 随后读取 x，却看到 0
```

如果 `T2` 确实发生在 `T1` 成功返回之后，那么这已经违反线性一致性，因为系统无法再被解释成“单副本按真实时间执行”。

## 与相邻概念的区别
- [[顺序一致性]]：只要求所有操作能排成一个全局顺序，不要求这个顺序尊重真实时间先后。
- [[最终一致性]]：只保证副本最终收敛，不保证写成功后立刻可见。

## 与工程实践的关系
- 对设计的影响：适合锁、配置、元数据、权限、账务等“宁可拒绝也不能读旧”的状态。
- 对实现的影响：常依赖 quorum、Leader 读、ReadIndex、共识日志提交等机制。
- 对排障的影响：一旦线上出现“写成功后立刻读旧”，要先检查副本延迟、读路径是否绕过 leader、读缓存是否过期。

## 最小例子

```text
T1: write x = 1  -> success
T2: read x       -> must see 1, if T2 starts after T1 finishes
```

## 常见误解
- “强一致就是线性一致”：很多人口语里的“强一致”不严谨，理论上它至少要区分顺序一致和线性一致。
- “线性一致就是事务”：不是。线性一致不等于跨多步操作的原子性和隔离性。
- “只要写入走 Raft 就一定线性一致”：读路径如果设计错了，整体对外仍不线性一致。

## 我的理解
线性一致性的真正价值，不是“理论上最强”，而是它把系统重新压缩回“像单机一样思考”的模型里。代价也正因此很高，因为你必须持续证明“没有任何人会在写成功后读到旧值”。
