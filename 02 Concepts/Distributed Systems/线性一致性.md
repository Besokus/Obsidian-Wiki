# 线性一致性

## 一句话解释
线性一致性（Linearizability）要求系统看起来像只有一个数据副本，每个操作在真实时间的某个瞬间不可分割地生效——操作完成后，所有后续读取必须看到这个结果。

## 它解决什么问题
多副本系统中，一个客户端写入成功后，另一个客户端可能读到旧值。线性一致性消除这种不确定性。

- 正确性：写入成功后，后续所有读取可观察到该写入。
- 可理解性：系统行为等价于单副本，应用层不需要处理"中间态"。
- 性能：延迟高，需要同步复制、quorum 或共识协议。

## 什么时候用
- 分布式锁、配置中心、元数据系统（etcd、ZooKeeper）。
- 金融账务、余额、权限等"不能错"的核心状态。
- 需要精确顺序的关键控制面数据。
- 分布式事务中的协调者。

## 什么时候不用
- 点赞数、浏览量、搜索索引等可接受短暂不一致。
- 延迟敏感的读多写少场景。
- 跨地域多活的强一致性代价过高。

## 核心机制
```text
1. 写入在副本间同步复制或通过共识（Raft/Paxos）确认。
2. 读取追齐最新版本（读 Leader、quorum 读、等待确认）。
3. 操作在真实时间轴上有一个明确的生效点（linearization point）。
4. 所有客户端看到该生效点之后的效果。
```

线性一致性不要求单个原子事务覆盖多个 key——它关心的是单个操作在时间上的原子可见性。

## 开发者视角
- 线性一致 ≠ 完全正确：线性一致保证每个操作及时生效，但不保证跨操作的事务性（ACID）。
- 线性一致是有代价的：网络分区时可能拒绝请求（CP 选择）。
- 在共识系统中（etcd/Raft），写入确认通常对应线性一致语义，但读取策略需要额外配置（读 Leader、quorum 读、等待 committed index）。
- 常见误区：认为"强一致性"就是线性一致。实际上顺序一致性也是强一致，但并不等价于线性一致。

## 工程判断
- 判断信号：客户端读到旧值导致业务异常、分布式锁超时后出现并发写入、配置更新后有的客户端看不到新值。
- 关键权衡：线性一致 vs 高性能高可用：通常二选一或做出明确的 CAP 取舍。
- 常见替代方案：使用共识协议（Raft/Paxos）、使用分布式协调服务（etcd/ZK）代理强一致操作。

## 最小例子
```text
T1: write x = 1  → 成功
T2: read x       → 必须看到 1（如果 T2 在 T1 之后发生）
```

## 常见误解
- "强一致就是线性一致"：顺序一致性也是强一致，但不依赖真实时间。
- "线性一致就是事务"：线性一致只保证单操作的原子可见性，不是跨操作的事务隔离。

## 易混概念
- [[顺序一致性]]：不要求操作按真实时间生效，只要求一个全局顺序且各客户端的程序顺序不矛盾。
- [[最终一致性]]：不保证任何时间点的实时可见性，只保证最终收敛。
- [[一致性模型]]：一致性模型的总览谱系。

## 我的理解
线性一致性是分布式系统中最强、也最贵的语义。它的价值在于消除"读到旧值"的心智负担，代价是遇到分区时必须选择拒绝请求。选不选线性一致，本质上是在问：这个功能的写入错误成本高，还是不可用的成本高。
