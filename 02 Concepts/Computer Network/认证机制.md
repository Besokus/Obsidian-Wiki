# 认证机制：Cookie / Session / Token

## Cookie

- 存在**客户端**，由浏览器自动携带
- 每次 HTTP 请求自动发送给服务器
- 常用于保存用户标识或偏好
- 大小有限，相对不安全

## Session

- 存在**服务器端**
- 浏览器通过 SessionID（通常在 Cookie 中）识别用户
- 相对安全
- **有状态**：服务器需要保存会话信息

## Token（常见为 JWT）

- 服务器签发的"访问凭证"
- 包含用户身份和权限信息
- **无状态**：服务器不存 Session，任意服务器都能校验
- 天然支持负载均衡，适合分布式、微服务、前后端分离

### Token 工作流程

```text
登录：用户提交凭证 → 服务器验证 → 签发 Token → 返给客户端
请求：客户端携带 Token（Authorization: Bearer <token>）
校验：服务器解析 → 校验签名 → 校验过期 → 识别用户身份
```

### Access Token + Refresh Token

| Token | 有效期 | 用途 |
|---|---|---|
| Access Token | 短（如 15 分钟） | 访问受保护资源 |
| Refresh Token | 长（如 7-30 天） | 获取新的 Access Token |

"记住我"功能：Refresh Token 存更长时间。

### 为什么 Token 能做到"无状态"

用户身份信息已包含在 Token 中，服务端只需验证签名即可识别用户，无需保存会话状态。

---

## 对比总结

| | Cookie | Session | Token |
|---|---|---|---|
| 存储位置 | 客户端 | 服务端 | 客户端 |
| 状态 | 有状态 | 有状态 | 无状态 |
| 扩展性 | 差 | 差（需共享 Session） | 好（天然支持分布式） |
| 安全性 | 低 | 较高 | 高 |
