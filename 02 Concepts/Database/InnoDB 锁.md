# InnoDB 锁

## 一句话解释
InnoDB 锁用于协调并发事务对数据的访问，保证读写正确性。锁粒度越小，并发越好，但理解和排查成本也越高。

## 它回答的核心问题
- 它试图解释：为什么并发更新同一行时不会互相踩踏。
- 它试图约束：哪些读写要互斥，哪些可以并行。
- 它试图区分：快照读、当前读、表锁、行锁、间隙锁分别在干什么。

## 它解决什么问题
- 防止多个事务同时修改同一数据造成冲突。
- 配合 [[MySQL 事务隔离级别]] 避免并发读写异常。
- 在可重复读下通过 Next-Key Lock 等机制减少幻读。

## 锁的层级
- 全局锁：锁整个数据库实例，典型场景是全库逻辑备份。
- 表级锁：锁整张表，包括表锁、元数据锁、意向锁。
- 行级锁：锁具体记录或索引范围，包括 Record Lock、Gap Lock、Next-Key Lock。

## 核心机制
```text
1. 普通 select 通常不加锁，属于快照读。
2. update/delete/insert 会自动加排他锁。
3. select ... for update 会加排他锁。
4. select ... lock in share mode 会加共享锁。
5. InnoDB 行锁是加在索引上的，不走索引可能扩大锁范围。
```

## 重点部分详解
- 机制细节：InnoDB 很多锁不是“锁整张表”，而是锁索引记录或索引范围。
- 常见故障：不走索引的更新可能扩大锁范围，导致大量等待。
- 典型代价：锁持有时间越长，并发越差，死锁概率也越高。
- 实战判断：慢更新不一定是 SQL 慢，也可能是锁等待；排查时要同时看执行计划和锁信息。

## 开发者视角
- 慢更新不一定是 SQL 慢，也可能是锁等待。
- 事务越长，锁持有时间越长。
- 更新条件必须尽量命中索引，否则可能锁住大量记录。
- DDL 可能被 MDL 阻塞，线上改表要谨慎。
