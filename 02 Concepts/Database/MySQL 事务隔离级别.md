# MySQL 事务隔离级别

## 一句话解释
事务隔离级别定义并发事务之间能看到彼此修改的程度。隔离越强，数据异常越少，并发能力通常越低。

## 它回答的核心问题
- 它试图解释：并发事务之间到底能看到什么、不能看到什么。
- 它试图约束：脏读、不可重复读、幻读分别在什么语义下出现或被禁止。
- 它试图区分：隔离级别是语义选择，不是性能调参按钮。

## 它解决什么问题
- 脏读：读到其他事务未提交的数据。
- 不可重复读：同一事务内两次读取同一行，结果不一致。
- 幻读：同一事务内按条件查询，前后出现新匹配行。

## 重点部分详解
- 核心边界：隔离级别描述的是“可见性规则”，不是“绝对不会并发冲突”。
- Read Uncommitted：会看到未提交修改，异常最多，实际很少用于业务场景。
- Read Committed：每次读都生成新 ReadView，更接近“读已提交”语义。
- Repeatable Read：事务内复用 ReadView，读到的视图更稳定，是 MySQL 默认隔离级别。
- Serializable：通过更强的并发控制换取最强一致性，代价通常最高。
- 实战判断：如果业务只需要避免脏读，不必盲目上到最高隔离级别；要根据一致性要求和并发压力折中。

## 核心机制
| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|---|---|---|---|---|
| Read Uncommitted | 可能 | 可能 | 可能 | 性能高，但安全性差 |
| Read Committed | 不会 | 可能 | 可能 | 每次读看到已提交版本 |
| Repeatable Read | 不会 | 不会 | 理论可能，InnoDB 常见场景下由 MVCC/锁处理 | MySQL 默认 |
| Serializable | 不会 | 不会 | 不会 | 隔离最强，并发最低 |

## 开发者视角
- 查看当前隔离级别：`SELECT @@transaction_isolation;`
- 设置隔离级别：`SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;`
- MySQL 默认使用 Repeatable Read，要结合 [[MVCC]] 和 [[InnoDB 锁]] 理解。
- 隔离级别不是越高越好，核心是业务能接受什么一致性与性能权衡。

## 使用场景
- 账户、库存等强一致写入场景，需要明确事务边界和锁行为。
- 报表、读多写少场景，可能更关注读性能和可接受的数据新鲜度。

## 常见误解
- 隔离级别越高越好。
- 只要用 Repeatable Read 就不会有任何并发问题。
- 幻读只靠 MVCC 就能完全解决。

## 易混概念
- [[MVCC]]：减少读写冲突的实现机制。
- [[InnoDB 锁]]：当前读和写操作的重要隔离手段。
- [[ACID]]：隔离级别对应 ACID 中的 Isolation。

## 我的理解
隔离级别是数据库把并发复杂度暴露给开发者的一个旋钮。写业务代码时，如果不清楚当前隔离级别，就很难判断一次读取到底是“当前最新值”还是“事务快照里的值”。
