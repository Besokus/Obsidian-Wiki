# Redis 在实际开发中的应用场景、高级用法与工程挑战

## 1. 先建立正确认知

Redis 不是"更快的 MySQL"，也不只是缓存。它本质上是一个**基于内存的数据结构服务器**，适合处理：

- 高频读写
- 临时状态
- 计数统计
- 分布式协调
- 排行榜
- 限流
- 消息流
- 缓存加速
- 低延迟数据访问

Redis 的核心优势是：

```text
内存存储 + 高效数据结构 + 原子命令 + TTL + 分布式能力
```

但 Redis 也有明显边界：

```text
不适合承担强一致、强事务、长期可靠存储的主数据职责
```

一句话总结：

> Redis 最适合作为高性能辅助存储层，而不是所有业务数据的最终事实来源。

对于更深入的理论基础，可参考 [[Redis为什么快]]、[[Redis的数据结构和使用场景]]。

## 2. Redis 在实际开发中的核心应用场景

### 2.1 缓存：最常见场景

#### 典型流程

```text
请求
  ↓
查 Redis
  ↓ 命中
直接返回
  ↓ 未命中
查数据库
  ↓
写入 Redis
  ↓
返回结果
```

例如：

```text
user:1001 -> 用户信息
product:2001 -> 商品详情
config:homepage -> 首页配置
permission:user:1001 -> 用户权限
```

#### 适合缓存的数据

适合放 Redis 的数据通常有几个特征：

- 读多写少
- 查询数据库成本高
- 允许短暂不一致
- 数据可以设置过期时间
- 热点访问明显

例如：

- 商品详情
- 用户基础信息
- 首页推荐
- 字典表
- 系统配置
- 权限菜单
- 热门文章
- 活动页数据

#### 不适合缓存的数据

下面这些数据要谨慎：

- 账户余额
- 支付状态
- 订单最终状态
- 库存最终账本
- 审计日志
- 法务/合规要求强持久化的数据

这些数据可以用 Redis 加速，但不能只存在 Redis。

---

### 2.2 Session / Token 存储

在单体应用中，Session 可以放在服务本机内存中。  
但在分布式系统中，请求会被负载均衡到不同机器：

```text
请求 1 -> Server A
请求 2 -> Server B
请求 3 -> Server C
```

如果 Session 只存在 Server A，其他机器就识别不了用户登录态。

Redis 可以作为统一登录态存储：

```text
login:token:abc123 -> userId=1001
TTL: 2h
```

常见用途：

- 登录 Session
- 短信验证码
- 邮箱验证码
- OAuth 临时 code
- JWT 黑名单
- 单点登录状态
- 临时授权 token

#### 为什么 Redis 适合？

因为 Redis 支持：

- 快速读写
- 自动过期
- 多服务共享
- 原子计数
- 简单删除失效

### 2.3 分布式锁

当多个服务实例同时操作同一资源时，需要互斥控制。

例如：

- 防止重复下单
- 防止重复支付
- 防止优惠券重复领取
- 防止定时任务多实例重复执行
- 防止多个线程同时刷新同一个缓存
- 防止库存并发扣减出现竞争

#### Redis 分布式锁基本写法

加锁：

```text
SET lock:order:1001 uniqueValue NX EX 10
```

含义：

- `NX`：key 不存在才设置成功
- `EX 10`：10 秒后自动过期
- `uniqueValue`：锁持有者标识，防止误删别人的锁

释放锁不能直接：

```text
DEL lock:order:1001
```

因为可能出现这种情况：

```text
线程 A 获得锁
线程 A 执行业务太久，锁过期
线程 B 获得新锁
线程 A 执行完，DEL 删除了线程 B 的锁
```

正确释放方式是：**判断 value 是自己的，再删除**。通常用 Lua 脚本保证原子性：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
  return redis.call("del", KEYS[1])
else
  return 0
end
```

#### 工程注意

Redis 分布式锁适合很多普通业务互斥场景，但不是绝对可靠的分布式一致性方案。

高可靠场景下，需要考虑：

- 锁过期时间如何设置？
- 业务没执行完锁过期怎么办？
- Redis 主从切换导致锁丢失怎么办？
- 客户端 GC pause 或网络抖动怎么办？
- 是否需要 watchdog 自动续期？
- 是否应该改用数据库锁、ZooKeeper、etcd？

相关的面试笔记：[[Redis 分布式锁怎么实现？]]、[[Redis 锁一定安全吗？有哪些风险？]]

### 2.4 接口限流

Redis 非常适合做分布式限流。

#### 场景

- 登录接口防爆破
- 短信验证码防刷
- API 网关限流
- 用户操作频控
- 秒杀入口限流
- 防止爬虫高频访问

#### 固定窗口限流

例如用户每分钟最多请求 100 次：

```text
rate:user:1001:202605141030 -> 57
TTL: 60s
```

每次请求：

```text
INCR rate:user:1001:minute
EXPIRE rate:user:1001:minute 60
```

超过 100 次就拒绝。

优点：简单。  
缺点：窗口边界可能出现突刺。

例如：

```text
10:00:59 请求 100 次
10:01:00 请求 100 次
```

实际 1 秒内可能通过 200 次。

---

#### 滑动窗口限流

用 ZSet 记录请求时间戳：

```text
ZADD rate:user:1001 1715670000000 req-1
ZREMRANGEBYSCORE rate:user:1001 0 now-60000
ZCARD rate:user:1001
```

流程：

```text
删除 60 秒前的请求记录
统计最近 60 秒请求数量
判断是否超过阈值
写入当前请求
```

优点：更平滑。  
缺点：每次请求都要维护 ZSet，成本更高。

---

#### 令牌桶 / 漏桶

更高级的限流方式可以用 Redis + Lua 实现。

适合：

- API 网关
- 高价值接口保护
- 高并发秒杀入口
- 多服务共享限流状态

### 2.5 排行榜

Redis 的 ZSet 非常适合做排行榜。

ZSet 每个元素都有一个 score，可以按 score 排序。

#### 示例：游戏积分榜

```text
ZADD rank:game 9800 user:1001
ZADD rank:game 8700 user:1002
ZREVRANGE rank:game 0 9 WITHSCORES
```

常见需求：

- 获取 Top 10
- 获取用户当前排名
- 更新用户分数
- 获取某个分数区间的用户
- 日榜、周榜、月榜、总榜

#### Key 设计

```text
rank:daily:2026-05-14
rank:weekly:2026-W20
rank:monthly:2026-05
rank:total
```

#### 工程注意

排行榜容易出现几个问题：

- 榜单数据过大
- 热门榜单访问极高
- 历史榜单是否要归档
- 分数更新是否需要落库
- Top N 是否可以和全量榜单拆开存储

如果只展示 Top 1000，就没必要把所有用户都长期放进一个巨大 ZSet。

### 2.6 计数器

Redis 的 `INCR`、`DECR`、`INCRBY` 是原子操作，非常适合计数。

#### 常见场景

- 文章阅读数
- 点赞数
- 收藏数
- 视频播放数
- 接口调用次数
- 用户签到次数
- 库存预扣
- 页面访问 PV

示例：

```text
INCR article:1001:view
INCR video:2001:play
INCRBY product:3001:stock -1
```

#### 工程问题

计数器看起来简单，但要想清楚：

1. Redis 里的计数是否需要同步到数据库？
2. 允许丢失一部分计数吗？
3. 是否需要定时批量落库？
4. 是否需要防止重复计数？
5. 是否需要按天、按小时拆分？

例如阅读量丢一点可以接受，余额变更绝对不能只用 Redis 计数。

### 2.7 秒杀 / 库存预扣

秒杀场景中，数据库通常扛不住瞬时高并发扣库存。Redis 可以作为前置库存层。

#### 基本流程

```text
活动开始前
  ↓
库存加载到 Redis

用户请求秒杀
  ↓
Redis 原子扣减库存
  ↓ 成功
写入消息队列
  ↓
异步创建订单
  ↓
数据库最终落库
```

#### 示例

```text
seckill:product:1001:stock -> 100
```

扣减：

```text
DECR seckill:product:1001:stock
```

但实际不能只用简单 `DECR`，因为要防止扣成负数。通常用 Lua：

```lua
local stock = tonumber(redis.call("get", KEYS[1]))
if stock == nil or stock <= 0 then
  return 0
end
redis.call("decr", KEYS[1])
return 1
```

#### 秒杀系统中的 Redis 价值

- 把高并发请求挡在数据库前
- 原子扣减库存
- 快速判断是否售罄
- 限制用户重复下单
- 支持队列化异步落库

#### 挑战

- Redis 扣减成功但订单创建失败怎么办？
- 用户重复请求如何幂等？
- Redis 库存和数据库库存如何对账？
- MQ 消费失败如何补偿？
- 超卖和少卖如何处理？

### 2.8 消息队列

Redis 可以做轻量级消息队列。

#### List 队列

```text
LPUSH queue:email task1
BRPOP queue:email 0
```

适合简单异步任务：

- 发邮件
- 发短信
- 异步写日志
- 简单任务分发

缺点：

- 消费确认能力弱
- 消息重试机制弱
- 可观测性较差
- 不适合大规模消息堆积

---

#### Stream 消息流

Redis Stream 是更完整的消息结构，支持：

- 消息 ID
- 消费者组
- 消息确认
- Pending List
- 消息回溯
- 多消费者协作

示例：

```text
XADD stream:order * orderId 1001 status paid
XREADGROUP GROUP group1 consumer1 STREAMS stream:order >
XACK stream:order group1 messageId
```

适合：

- 轻量事件流
- 订单状态通知
- 异步任务
- 日志流
- 小规模消息队列

#### Redis Stream 和 Kafka 的区别

| 维度 | Redis Stream | Kafka |
|---|---|---|
| 定位 | 轻量消息流 | 分布式日志系统 |
| 存储 | 内存为主，可持久化 | 磁盘顺序日志 |
| 堆积能力 | 一般 | 很强 |
| 消费生态 | 简单 | 成熟 |
| 适合场景 | 中小规模异步任务 | 大规模事件流、日志、数据管道 |

关键业务、大规模堆积、长期消息保留，一般更推荐 Kafka、RocketMQ、Pulsar。

---

### 2.9 延迟队列

Redis ZSet 可以实现延迟任务。

#### 原理

把任务执行时间作为 score：

```text
ZADD delay:queue 1715673600000 order:1001
```

消费者扫描到期任务：

```text
ZRANGEBYSCORE delay:queue 0 now LIMIT 0 1
```

如果任务到期，就取出执行并删除。

#### 场景

- 订单 30 分钟未支付自动取消
- 优惠券到期提醒
- 定时发送通知
- 超时关闭工单
- 延迟重试任务

#### 工程挑战

- 多消费者抢同一任务
- 删除和执行要保证一致性
- 执行失败要重试
- 大量任务会造成 ZSet 压力
- 轮询频率影响延迟和资源消耗

通常需要 Lua 脚本保证"取出 + 删除"的原子性。

### 2.10 布隆过滤器

布隆过滤器用于判断一个元素是否可能存在。

特点：

```text
判断不存在：一定不存在
判断存在：可能存在
```

它有误判率，但空间效率极高。

#### 场景

- 防止缓存穿透
- 用户 ID 是否存在
- 商品 ID 是否存在
- URL 去重
- 黑名单过滤
- 爬虫去重

#### 解决缓存穿透

没有布隆过滤器时：

```text
请求不存在的 userId
  ↓
Redis 查不到
  ↓
DB 查不到
  ↓
大量无效请求打爆 DB
```

有布隆过滤器后：

```text
请求 userId
  ↓
BloomFilter 判断一定不存在
  ↓
直接拒绝，不查 Redis / DB
```

Redis 可以通过 RedisBloom 模块支持布隆过滤器。

### 2.11 Bitmap 位图

Bitmap 用一个 bit 表示一个状态，非常节省空间。

#### 用户签到

```text
SETBIT sign:user:1001:2026-05 13 1
GETBIT sign:user:1001:2026-05 13
BITCOUNT sign:user:1001:2026-05
```

表示用户 1001 在 5 月第 14 天是否签到。

#### 适合场景

- 用户签到
- 用户是否活跃
- 是否已读
- 是否领取奖励
- 大规模布尔状态统计

优点：

- 极省内存
- 位操作快
- 适合海量布尔状态

缺点：

- 只能表达 0/1 状态
- 不适合存复杂对象
- 稀疏 ID 可能浪费空间

### 2.12 HyperLogLog 去重统计

HyperLogLog 用于估算去重数量。

#### UV 统计

```text
PFADD uv:2026-05-14 user1001
PFADD uv:2026-05-14 user1002
PFCOUNT uv:2026-05-14
```

适合：

- UV
- 活动参与人数
- 搜索去重人数
- 页面访问去重用户数

优点：

- 内存占用极低
- 适合大规模去重计数

缺点：

- 是近似值
- 无法取出具体用户列表
- 不适合强精确统计

### 2.13 Geo 地理位置

Redis Geo 支持经纬度存储和附近查询。

#### 示例

```text
GEOADD shop:geo 121.4737 31.2304 shop:1001
GEORADIUS shop:geo 121.47 31.23 5 km
```

适合：

- 附近门店
- 附近的人
- 附近车辆
- 外卖配送范围
- 打车派单候选

#### 注意

Redis Geo 适合简单附近查询。  
如果需要复杂地图路径规划、行政区划、多边形范围查询，应该使用专业 GIS 能力。

### 2.14 发布订阅 Pub/Sub

Redis Pub/Sub 可以实现轻量广播：

```text
SUBSCRIBE channel:config
PUBLISH channel:config "reload"
```

适合：

- 配置变更通知
- 本地缓存失效通知
- 简单服务广播
- WebSocket 网关消息转发

但它有明显缺点：

- 不持久化
- 消费者离线会丢消息
- 没有 ACK
- 没有重试
- 不适合关键业务消息

关键业务消息应使用 Stream 或专业 MQ。

## 3. Redis 高级用法

### 3.1 Lua 脚本

Lua 是 Redis 高级用法中非常重要的一类能力。

#### 作用

把多个 Redis 命令合并成一个原子操作。

例如"判断库存大于 0，然后扣减库存"：

```lua
local stock = tonumber(redis.call("get", KEYS[1]))
if stock == nil or stock <= 0 then
  return 0
end
redis.call("decr", KEYS[1])
return 1
```

#### 常见用途

- 分布式锁解锁
- 秒杀扣库存
- 限流
- 延迟队列抢任务
- 多 key 条件判断
- 幂等控制

#### 注意

Lua 脚本执行期间会阻塞 Redis 主线程，所以不能写复杂长逻辑。

---

### 3.2 Pipeline

Pipeline 用来减少网络往返。

普通请求：

```text
发送命令 1 -> 等返回
发送命令 2 -> 等返回
发送命令 3 -> 等返回
```

Pipeline：

```text
一次发送命令 1、2、3
一次接收结果 1、2、3
```

适合：

- 批量写缓存
- 批量查询
- 批量删除
- 批量初始化数据
- 批量计数同步

注意：

> Pipeline 不是事务，只是网络优化。

---

### 3.3 MULTI / EXEC 事务

Redis 支持简单事务：

```text
MULTI
INCR a
DECR b
EXEC
```

特点：

- 命令先排队
- EXEC 时顺序执行
- 中间不会插入其他客户端命令
- 但不支持传统数据库那种完整回滚

适合简单原子批处理，不适合复杂业务事务。

---

### 3.4 WATCH 乐观锁

`WATCH` 可以监控 key 是否被修改。

典型流程：

```text
WATCH stock:1001
GET stock:1001
MULTI
DECR stock:1001
EXEC
```

如果 `EXEC` 前 `stock:1001` 被别人改了，事务失败。

适合：

- 简单 CAS
- 条件更新
- 并发冲突不高的场景

如果冲突很高，WATCH 会大量失败，性能不好。

---

### 3.5 Scan 系列命令

生产环境不要随便用：

```text
KEYS *
```

因为它会一次性扫描所有 key，可能阻塞 Redis。

应该使用：

```text
SCAN
HSCAN
SSCAN
ZSCAN
```

它们是渐进式扫描，适合：

- 后台清理
- 数据迁移
- 运维排查
- 批量统计

---

### 3.6 Hash Tag 控制 Cluster 槽位

Redis Cluster 中，不同 key 会落到不同 slot。  
如果多 key 命令涉及不同 slot，会失败。

例如：

```text
MGET user:1:name user:1:age
```

如果两个 key 不在同一 slot，Cluster 下可能无法执行。

可以使用 hash tag：

```text
user:{1}:name
user:{1}:age
user:{1}:profile
```

`{1}` 内部的内容会作为 hash 依据，这些 key 会落到同一个 slot。

适合：

- 同用户相关 key
- 同订单相关 key
- 需要多 key 操作的数据

---

### 3.7 Redisson 高级能力

Java 开发中常用 Redisson，它封装了很多 Redis 分布式对象：

- 分布式锁
- 可重入锁
- 公平锁
- 读写锁
- 信号量
- 闭锁 CountDownLatch
- 分布式集合
- 延迟队列
- 看门狗续期

例如 Redisson 分布式锁比手写 `SET NX EX` 更安全一些，因为它处理了：

- 唯一标识
- Lua 解锁
- 自动续期
- 可重入
- 异常释放

但不要误以为 Redisson 能解决所有一致性问题。底层仍受 Redis 架构限制。

### 3.8 多级缓存

典型结构：

```text
本地缓存 Caffeine / Guava
        ↓
Redis
        ↓
数据库
```

#### 为什么需要多级缓存？

Redis 虽然快，但仍然有网络开销。  
本地缓存直接在应用进程内，延迟更低。

适合：

- 极热点配置
- 权限数据
- 首页热点数据
- 高频字典数据

#### 难点

- 本地缓存如何失效？
- 多台机器本地缓存如何同步？
- Redis 更新后本地缓存是否会读到旧值？
- 是否通过 Pub/Sub 通知各节点删除本地缓存？

---

### 3.9 缓存预热

系统刚上线或 Redis 刚重启时，缓存为空。  
如果大量请求直接打到数据库，会造成冲击。

缓存预热指提前加载热点数据：

```text
应用启动 / 定时任务 / 发布流程
  ↓
查询热点数据
  ↓
写入 Redis
```

适合：

- 秒杀商品
- 首页配置
- 热门榜单
- 活动数据
- 字典数据

---

### 3.10 逻辑过期

普通 TTL 是 Redis 自动删除 key。  
逻辑过期是把过期时间作为 value 的一部分。

例如：

```json
{
  "data": {
    "id": 1001,
    "name": "商品 A"
  },
  "expireAt": 1715673600000
}
```

读取时：

```text
发现未过期 -> 直接返回
发现已过期 -> 先返回旧值，同时异步刷新
```

适合热点数据，尤其是防止缓存击穿。

优点：

- 不会因为 key 物理过期导致瞬间所有请求打到数据库
- 可以平滑刷新缓存

缺点：

- 可能返回旧数据
- 逻辑复杂
- 需要异步刷新和锁控制

---

### 3.11 异步刷新缓存

对于不要求强实时的数据，可以后台刷新：

```text
定时任务 / MQ 消费 / binlog 监听
  ↓
刷新 Redis
```

适合：

- 榜单
- 首页模块
- 推荐结果
- 统计指标
- 热点商品

请求路径只读 Redis，性能很高。代价是数据实时性下降。

---

## 4. 使用 Redis 面临的工程挑战

### 4.1 缓存一致性

这是 Redis 最核心的工程难题之一。

#### 问题

数据库更新了，但 Redis 里还是旧数据。

```text
DB: name = "Bob"
Redis: name = "Alice"
```

用户可能读到旧值。

---

#### 常用方案一：先更新数据库，再删除缓存

```text
更新 DB
删除 Redis
```

这是实际项目中最常见的方案。

为什么不是"更新 DB 后更新缓存"？

因为并发情况下可能出现旧数据覆盖新缓存。

例如：

```text
线程 A 更新 DB 为 v1
线程 B 更新 DB 为 v2
线程 B 更新缓存为 v2
线程 A 更新缓存为 v1
```

最后缓存反而是旧值。

删除缓存更稳妥。下次读取时重新从 DB 加载。

---

#### 常用方案二：延迟双删

```text
删除缓存
更新数据库
等待几十到几百毫秒
再次删除缓存
```

或者：

```text
更新数据库
删除缓存
延迟后再次删除缓存
```

目的是降低并发读写导致脏缓存的概率。

但它不是绝对强一致，只是工程补偿。

---

#### 常用方案三：MQ 异步删除缓存

```text
业务更新 DB
发送缓存失效消息
消费者删除 Redis
失败则重试
```

优点：

- 解耦业务逻辑
- 支持失败重试
- 适合多缓存副本失效

缺点：

- 引入 MQ 复杂度
- 需要保证消息可靠性
- 需要处理重复消费和幂等

---

#### 常用方案四：监听 binlog

通过监听 MySQL binlog 感知数据变化，然后刷新或删除 Redis。

典型架构：

```text
MySQL binlog
  ↓
Canal / Debezium
  ↓
缓存同步服务
  ↓
Redis 删除或更新
```

适合：

- 多服务共享缓存
- 业务不想感知缓存刷新
- 缓存同步逻辑希望集中治理

详细内容可参考 [[Redis 和数据库如何保持一致性]]。

---

#### 关键结论

缓存和数据库之间通常只能做到：

```text
最终一致
```

如果业务要求强一致，不应该依赖缓存作为判断依据。

例如：

- 支付是否成功：查 DB
- 余额是否足够：查 DB 或强事务系统
- 库存最终扣减：DB/MQ/事务保障
- 订单最终状态：DB 为准

---

### 4.2 缓存穿透

#### 问题

请求查询不存在的数据：

```text
Redis 没有
DB 也没有
```

如果攻击者大量请求不存在 ID，数据库会被打爆。

#### 解决方案

##### 1. 缓存空值

```text
user:999999 -> null
TTL: 60s
```

优点：简单。  
缺点：大量不存在 key 会占用 Redis 内存。

##### 2. 布隆过滤器

先判断 ID 是否可能存在。

```text
BloomFilter 判断不存在 -> 直接拒绝
BloomFilter 判断可能存在 -> 查 Redis / DB
```

优点：节省空间。  
缺点：有误判率，且删除元素复杂。

##### 3. 参数校验

例如：

- ID 小于 0 直接拒绝
- 非法格式直接拒绝
- 不符合业务规则直接拒绝

##### 4. 风控限流

对异常用户、IP、设备进行限流。

---

### 4.3 缓存击穿

#### 问题

某个热点 key 过期，瞬间大量请求都查不到 Redis，然后同时打到数据库。

```text
hot_product:1001 过期
10000 个请求同时访问
全部打到 DB
```

#### 解决方案

##### 1. 互斥锁重建缓存

只有一个线程查 DB 并重建缓存，其他线程等待或返回旧值。

```text
请求发现缓存不存在
  ↓
尝试获取 rebuild lock
  ↓ 成功
查 DB，写 Redis
  ↓ 失败
等待 / 返回旧数据 / 降级
```

##### 2. 逻辑过期

热点 key 不物理删除，而是内部带过期时间。  
过期后仍返回旧值，并异步刷新。

##### 3. 热点数据不过期

对极热点数据设置较长过期时间，靠后台任务刷新。

---

### 4.4 缓存雪崩

#### 问题

大量缓存同一时间失效，或者 Redis 集群故障，导致请求集中打到数据库。

#### 解决方案

##### 1. TTL 加随机值

```text
TTL = 3600 + random(0, 300)
```

避免大量 key 同时过期。

##### 2. 缓存预热

上线前提前加载热点数据。

##### 3. 多级缓存

```text
本地缓存 -> Redis -> DB
```

##### 4. 限流降级

当 Redis 或 DB 压力过高时：

- 返回默认值
- 返回旧值
- 暂停部分非核心功能
- 限制部分用户访问

##### 5. Redis 高可用

使用：

- 主从复制
- Sentinel
- Cluster
- 云厂商高可用 Redis

---

### 4.5 热 Key 问题

#### 问题

某个 key 被极高频访问，导致单个 Redis 节点压力过大。

例如：

- 秒杀商品详情
- 热门新闻
- 明星微博
- 爆款直播间
- 热门配置

#### 解决方案

##### 1. 本地缓存

应用实例内缓存热点数据，减少 Redis 请求。

##### 2. 热 Key 拆分

复制多个副本：

```text
hot:product:1001:0
hot:product:1001:1
hot:product:1001:2
```

读取时随机访问一个副本。

##### 3. 读写分离

读请求走从节点，但要注意复制延迟。

##### 4. 热点识别

通过：

- Redis monitor
- slowlog
- 客户端埋点
- 代理层统计
- 业务访问日志

识别热点 key。

---

### 4.6 大 Key 问题

#### 什么是大 Key？

例如：

- String value 几 MB
- Hash 有几十万个 field
- List 有百万个元素
- Set / ZSet 成员极多

#### 危害

- 网络传输慢
- 单命令执行慢
- 阻塞 Redis 主线程
- 删除慢
- 持久化压力大
- 主从同步压力大
- Cluster 数据倾斜

#### 解决方案

##### 1. 拆分 key

```text
user:1001:profile
user:1001:orders:page:1
user:1001:orders:page:2
```

##### 2. 控制集合长度

例如只保留最近 1000 条：

```text
LTRIM list:key 0 999
```

##### 3. 使用 UNLINK 删除

```text
UNLINK big:key
```

`UNLINK` 是异步删除，比 `DEL` 更不容易阻塞。

##### 4. 避免全量读取

不要随便：

```text
HGETALL big:hash
SMEMBERS big:set
LRANGE big:list 0 -1
```

要分页、分批、限量读取。

---

### 4.7 内存管理挑战

Redis 是内存数据库，内存是核心成本。

#### 常见问题

- 内存持续上涨
- key 没有 TTL
- 大量冷数据留在 Redis
- value 过大
- 内存碎片率高
- 淘汰策略不合理

#### 淘汰策略

常见策略：

| 策略 | 含义 |
|---|---|
| noeviction | 不淘汰，写入时报错 |
| allkeys-lru | 所有 key 中淘汰最近最少使用 |
| volatile-lru | 只淘汰设置了过期时间的 key |
| allkeys-lfu | 淘汰低频访问 key |
| volatile-ttl | 优先淘汰快过期的 key |

缓存型 Redis 常用：

```text
allkeys-lru
allkeys-lfu
```

如果 Redis 存的是重要业务状态，不能随便淘汰，要谨慎配置。

---

### 4.8 持久化风险

Redis 支持 RDB 和 AOF。可参考 [[Redis的如何实现持久化]]。

#### RDB

快照持久化。

优点：

- 文件紧凑
- 恢复快
- 适合备份

缺点：

- 快照之间的数据可能丢失
- fork 时可能有内存压力

#### AOF

记录写命令日志。

优点：

- 数据丢失更少
- 可配置刷盘策略
- 可通过 rewrite 压缩

缺点：

- 文件更大
- 恢复可能较慢
- 高频写入时磁盘压力更高

#### 工程判断

如果 Redis 只是缓存：

```text
可以接受丢失，靠 DB 重建
```

如果 Redis 存业务状态：

```text
必须认真配置 AOF、备份、主从、故障恢复
```

但仍然要注意：Redis 不应轻易替代强一致数据库。

---

### 4.9 主从复制与故障切换

Redis 主从复制通常是异步的。

#### 风险

```text
客户端写入 Master
Master 还没同步给 Replica
Master 宕机
Replica 升级为新 Master
刚才写入的数据丢失
```

所以 Redis 主从高可用并不等于零丢失。

#### Sentinel

Sentinel 可以监控 Redis 主节点，并在主节点故障时自动切换。

适合：

- 单主多从
- 自动故障转移
- 中小规模 Redis 高可用

#### Cluster

Cluster 支持分片和高可用。

适合：

- 数据量大
- 单机内存不够
- QPS 很高
- 需要水平扩展

但 Cluster 也带来复杂度：

- slot 迁移
- 跨 slot 操作限制
- 客户端重定向
- 数据倾斜
- 热点 key
- 扩容缩容成本

---

### 4.10 慢命令与阻塞

Redis 核心命令执行通常是单线程。  
一个慢命令可能阻塞后续所有请求。

危险命令：

```text
KEYS *
HGETALL big:hash
SMEMBERS big:set
LRANGE big:list 0 -1
DEL big:key
长时间 Lua 脚本
```

替代方案：

```text
SCAN 替代 KEYS
分页读取替代全量读取
UNLINK 替代 DEL
拆分大 key
控制 Lua 脚本复杂度
```

## 5. Redis 典型架构设计

```text
Client
  ↓

  ↓
Redis
  ↓ miss
Database
```

适合大多数业务缓存。

### 5.2 多级缓存架构

```text
Client
  ↓
App
  ↓
Local Cache
  ↓ miss

  ↓ miss
```

适合极热点读场景。

### 5.3 秒杀架构

```text
Client
  ↓
网关限流
  ↓
Redis 判断库存 / 去重
  ↓
MQ
  ↓
订单服务异步创建订单
```

Redis 在这里负责：

```text
  ↓

```

适合复杂系统中的统一缓存治理。

## 6. 实际开发中的 Key 设计建议

### 6.1 命名规范

推荐格式：

```text
业务:实体:ID:属性
```

例如：

```text
user:1001:profile
user:1001:permission
product:2001:detail
order:3001:status
rank:daily:2026-05-14
```

### 6.2 避免 key 太长

key 太长会浪费内存。  
但也不要短到不可维护。

不好：

```text
u:1:p
```

太难维护。

较好：

```text
user:1:profile
```

### 6.3 合理设置 TTL

缓存类 key 尽量设置 TTL：

```text
EXPIRE product:1001:detail 3600
```

同时避免所有 key 同时过期：

```text
TTL = baseTTL + random
```

### 6.4 区分缓存 key 和状态 key

缓存 key 可以被删除、过期、重建。  
状态 key 可能涉及业务正确性，要更谨慎。

例如：

```text
cache:product:1001
state:order:1001:processing
lock:order:1001
rate:user:1001
```

## 7. Redis 使用决策框架

当你想把某个数据放进 Redis 时，先问 8 个问题：

1. 这个数据是否高频访问？
2. 这个数据是否可以短暂不一致？
3. 这个数据是否可以丢失或重建？
4. 是否有明确过期时间？
5. 查询模式是否主要按 key 访问？
6. 数据量是否会无限增长？
7. 是否存在热 key 或大 key 风险？
8. Redis 故障时系统是否能降级？

如果答案是：

```text
高频访问 + 可重建 + 可过期 + 按 key 查询
```

Redis 很适合。

如果答案是：

```text
强一致 + 不可丢 + 复杂查询 + 需要长期审计
```

就不应该把 Redis 当主存储。

## 8. 常见误区

### 误区一：Redis 一定比数据库快

Redis 通常快，但如果你用错命令，例如：

```text
HGETALL 巨大 Hash
SMEMBERS 巨大 Set
KEYS *
```

Redis 也会非常慢，甚至阻塞。

---

### 误区二：缓存能保证强一致

缓存的目标是提升性能，不是保证最终事实。  
强一致数据应以数据库或事务系统为准。

---

### 误区三：所有数据都应该加缓存

低频数据加缓存可能得不偿失：

- 增加一致性问题
- 增加内存成本
- 增加代码复杂度
- 命中率很低

缓存应该优先服务高频热点数据。

---

### 误区四：分布式锁只要 SETNX 就行

错误写法：

```text
SETNX lock key
EXPIRE lock 10
```

这不是原子操作。  
如果 `SETNX` 成功后服务宕机，`EXPIRE` 没执行，锁会永久存在。

正确写法：

```text
SET lock:key uniqueValue NX EX 10
```

---

### 误区五：Redis 可以替代 MQ

Redis 可以做轻量消息队列，但不能完全替代专业 MQ。  
尤其是大规模堆积、复杂消费、严格可靠性场景，还是应使用 Kafka、RocketMQ、Pulsar 等。

---

## 9. 一个典型业务例子：商品详情页缓存

### 场景

电商商品详情页访问量很高，直接查数据库压力大。

### Key 设计

```text
product:1001:detail
product:1001:stock
product:1001:price
```

### 读取流程

```text
查 Redis 商品详情
  ↓ 命中
返回
  ↓ 未命中
获取互斥锁
  ↓ 成功
查 DB -> 写 Redis -> 返回
  ↓ 失败
短暂等待 / 返回兜底数据
```

### 更新流程

```text
后台修改商品
  ↓
更新数据库
  ↓
删除 Redis 商品缓存
  ↓
下次请求重新加载
```

### 防护措施

- TTL 加随机值，防止雪崩
- 热点商品逻辑过期，防止击穿
- 本地缓存保护 Redis
- 商品 ID 用布隆过滤器防止穿透
- 慢查询和热 key 监控
- 限流防止恶意刷接口

## 10. 总结

Redis 在实际开发中的主要价值是：

```text
用极低延迟处理高频、临时、简单、可重建的数据访问
```

它最适合：

- 缓存
- 登录态
- 分布式锁
- 限流
- 排行榜
- 计数器
- 秒杀库存
- 延迟任务
- 轻量消息队列
- 布隆过滤器
- Bitmap
- HyperLogLog
- Geo 查询
- 多级缓存

但 Redis 的工程挑战也很明显：

- 缓存一致性
- 缓存穿透、击穿、雪崩
- 热 key
- 大 key
- 内存成本
- 持久化风险
- 主从复制数据丢失
- Cluster 运维复杂度
- 慢命令阻塞
- 分布式锁正确性
- 与数据库、MQ、搜索引擎的边界划分

真正会用 Redis，不是会写几个命令，而是能判断：

> 哪些数据适合 Redis，哪些不适合；哪些场景 Redis 是加速层，哪些场景 Redis 不能承担正确性责任。
