# ACID

## 一句话解释
ACID 是事务可靠性的四个目标：**原子性、一致性、隔离性、持久性**。它描述的是数据库事务应该对外呈现出的行为。

## 它回答的核心问题
- 它试图解释：为什么数据库事务可以被当成一个“可靠边界”。
- 它试图约束：事务提交前后，系统必须满足哪些语义。
- 它试图区分：原子性、一致性、隔离性、持久性分别解决什么问题。

## 定义

```text
ACID = 原子性 Atomicity + 一致性 Consistency + 隔离性 Isolation + 持久性 Durability
```

## 它解决什么问题
- 原子性：防止操作只执行一半。
- 一致性：防止事务结束后违反业务约束。
- 隔离性：防止并发事务互相干扰。
- 持久性：防止事务提交后因崩溃丢失。

## 模型与假设
- 系统模型：数据库是事务边界内的可靠执行体。
- 时间 / 顺序假设：事务提交前后应该有明确边界。
- 故障模型：崩溃、重启、部分写入、并发读写都要考虑。
- 观察视角：用户只关心事务前和事务后的状态，而不是中间过程。

## 核心性质 / 关键结论
- 原子性保证“要么全成，要么全不成”。
- 一致性保证事务结果不破坏约束。
- 隔离性保证并发事务互不干扰到不可接受的程度。
- 持久性保证提交结果不会轻易丢失。

## 核心机制
```text
1. 原子性依赖 Undo Log 回滚。
2. 持久性依赖 Redo Log 恢复。
3. 隔离性依赖锁和 MVCC。
4. 一致性依赖数据库机制和业务约束共同保证。
```

## 重点性质详解
- 关键性质：ACID 不是一个单一功能，而是四种不同目标的组合。
- 为什么重要：它告诉你事务语义到底依赖哪些底层机制。
- 反例 / 边界：数据库只能帮你兜住“数据库内部”的一致性，业务一致性仍要靠代码和约束。
- 和相邻概念的差别：ACID 是事务性质，不是具体实现；[[MySQL 事务]] 是操作边界。

## 开发者视角
- 不要把“一致性”只理解成数据库自动保证，业务约束也要通过代码、约束、事务边界共同维护。
- 隔离性越强，并发能力通常越低。
- 事务不是免费的，长事务会增加锁等待和版本保留成本。
