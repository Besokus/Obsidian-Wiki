# 索引失效

## 一句话解释
索引失效是指本来有机会用索引的查询，因为写法、类型转换、条件组合或优化器选择等原因，最终没有按预期走索引。

## 它回答的核心问题
- 它试图解释：为什么“明明建了索引”查询还是慢。
- 它试图约束：哪些 SQL 写法会让索引利用率下降。
- 它试图区分：真正失效、部分失效、以及优化器主动不选索引不是一回事。

## 常见失效场景

| 场景 | 示例 |
|---|---|
| 使用 `!=` 或 `<>` | `WHERE age != 20` |
| 对字段做运算或函数 | `WHERE SUBSTRING(name, 1, 3) = 'abc'` |
| 模糊查询以通配符开头 | `WHERE name LIKE '%abc'` |
| 数据类型不一致 | `WHERE varchar_col = 123`（类型隐式转换） |
| `OR` 条件中有未索引列 | `WHERE a=1 OR unindexed_b=2` |
| 违背最左匹配原则 | 联合索引 `(a,b,c)`，但跳过了 `a` 直接查 `b` |

## 最左匹配原则详解

对于联合索引 `(a, b, c)`：

| 查询条件 | 是否命中 |
|---|---|
| `WHERE a=1` | ✓ |
| `WHERE a=1 AND b=2` | ✓ |
| `WHERE a=1 AND b=2 AND c=3` | ✓ |
| `WHERE b=2` | ✗（跳过了 a） |
| `WHERE a=1 AND c=3` | 部分（a 命中，c 不命中） |

## 重点部分详解
- 机制细节：索引是否失效，很多时候不是“有没有索引”，而是“条件能不能让索引有序地向下匹配”。
- 常见故障：函数包裹列、隐式类型转换、左模糊、跳过最左列，都是高频踩坑点。
- 典型代价：一旦走全表扫描，数据量越大越容易把查询拖慢。
- 实战判断：先看 `EXPLAIN`，再看条件写法，不要只盯着“有没有建索引”。

## 注意

- `OR` 本身不会导致索引必然失效，只要各条件都用索引，MySQL 可能使用 Index Merge
- 最终是否使用索引取决于优化器的成本评估
