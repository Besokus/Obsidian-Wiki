# 覆盖索引与回表

## 一句话解释
覆盖索引指查询需要的列都能从索引中直接拿到，不需要再回到聚簇索引取整行。回表是通过二级索引找到主键后，再查聚簇索引拿其他列。

## 它回答的核心问题
- 它试图解释：为什么“同样是走索引”，查询成本可能差很多。
- 它试图约束：哪些查询可以只读索引，不必访问整行。
- 它试图区分：二级索引命中和真正的覆盖索引不是一回事。

## 它解决什么问题
- 减少二次查表，降低随机 IO。
- 避免 `select *` 带来的不必要行数据读取。
- 提升高频查询的稳定性。

## 定义

```text
覆盖索引 = 查询所需字段都能从索引中直接拿到，不需要回表。
回表 = 通过二级索引拿到主键后，再回聚簇索引读取完整行。
```

## 什么时候用
- 高频查询只需要少量字段。
- 可以通过联合索引覆盖查询条件和返回列。
- 读性能比写入维护成本更重要。

## 什么时候不用
- 返回字段很多，为了覆盖索引把大量列都塞进联合索引，会让索引过大。
- 写入频繁的表要谨慎增加宽联合索引。

## 核心机制
```text
1. 二级索引叶子节点保存索引列和主键。
2. 如果查询列都在二级索引中，直接返回。
3. 如果查询需要其他列，通过主键回到聚簇索引查询整行。
```

## 重点部分详解
- 机制细节：二级索引叶子节点不存整行，只存索引列和主键；主键就是回表入口。
- 常见故障：`select *`、返回列过多、联合索引设计不合理，都会让覆盖索引失去意义。
- 典型代价：为了覆盖查询而把索引设计得太宽，会增加写入维护成本。
- 实战判断：如果 `EXPLAIN` 里出现 `Using index`，通常说明有机会走覆盖索引。

## 开发者视角
- 尽量避免无脑 `select *`。
- 用 [[EXPLAIN]] 的 Extra 字段观察是否出现 `Using index`。
- 覆盖索引经常和联合索引一起设计。

## 最小例子
```sql
-- 假设存在联合索引 (username, password)
select id, username, password from tb_user where username = 'itcast';
```

如果查询需要的列都在联合索引中，就可以减少回表。

## 我的理解
覆盖索引的核心是“少读一次”。它不是为了炫技，而是在高频查询里减少回表成本，尤其适合列表页、详情页中的固定字段查询。
