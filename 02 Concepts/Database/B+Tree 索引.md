# B+Tree 索引

## 一句话解释
B+Tree 是 MySQL 最常用的索引结构，适合磁盘场景下的范围查询、排序和有序扫描。它通过多路树降低树高，从而减少磁盘 IO 次数。

## 它回答的核心问题
- 它试图解释：为什么数据库索引要做成树，而不是简单数组或哈希表。
- 它试图约束：什么样的数据结构更适合磁盘页和范围扫描。
- 它试图区分：单点查找、范围查找、排序扫描分别需要什么样的索引能力。

## 它解决什么问题
- 二叉树或红黑树在大数据量下层级深，磁盘 IO 次数多。
- Hash 索引适合等值查询，但不适合范围查询和排序。
- 数据库需要一种既能定位单点数据，也能高效范围扫描的结构。

## 定义

```text
B+Tree = 一种多路平衡查找树，非叶子节点保存索引键，叶子节点保存数据或数据定位信息，并按顺序链接。
```

## 模型与假设
- 系统模型：存储介质是磁盘或页式存储，随机 IO 昂贵。
- 时间 / 顺序假设：顺序扫描比频繁随机跳转更便宜。
- 故障模型：索引要能适应大数据量和频繁更新。
- 观察视角：把“查找成本”转化为“树高 + 页访问次数”。

## 核心性质 / 关键结论
- 多路树能显著降低树高。
- 叶子节点有序链接，天然适合范围查询。
- 非叶子节点不存完整数据，能提高每个页可容纳的 key 数量。
- B+Tree 很适合数据库这种“查找 + 范围 + 排序”混合负载。

## 核心机制
```text
1. 非叶子节点主要存储键和指针。
2. 数据集中出现在叶子节点。
3. 叶子节点按顺序连接，方便范围查询。
```

## 重点部分详解
- 机制细节：B+Tree 把数据集中放在叶子节点，非叶子只做导航，所以更适合磁盘页。
- 常见故障：主键过大、随机插入、页分裂都会放大写入成本。
- 典型代价：索引越宽，每页能放的 key 越少，树高和 IO 成本可能上升。
- 实战判断：如果查询经常要范围扫描、排序、前缀匹配，B+Tree 往往比 Hash 更合适。

## 开发者视角
- 范围查询、排序和最左前缀匹配都和 B+Tree 的有序性有关。
- 主键越大，索引页可容纳的 key 越少，树可能更高。
- 随机主键可能导致页分裂，影响写入性能。

## 为什么 InnoDB 常用 B+Tree
- 相比二叉树，树高更低，磁盘 IO 更少。
- 相比 B-Tree，非叶子节点不保存完整数据，一页能放更多 key。
- 相比 Hash，支持范围查询和排序。

## 和理论的关系
- 依赖的理论概念：有序性、平衡树、局部性原理
- 关键假设：磁盘页访问远慢于内存访问。
- 理论边界对工程的影响：索引设计本质上是在“查找成本”和“写入成本”之间做折中。
