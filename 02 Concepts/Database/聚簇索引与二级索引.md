# 聚簇索引与二级索引

## 一句话解释
聚簇索引的叶子节点直接保存整行数据，二级索引的叶子节点保存索引列和主键值。InnoDB 表的数据本身就是按聚簇索引组织的。

## 它回答的核心问题
- 它试图解释：为什么主键查询快，非主键查询有时要回表。
- 它试图约束：表数据在 InnoDB 里到底是怎么组织的。
- 它试图区分：聚簇索引、二级索引、回表三者的关系。

## 它解决什么问题
- 用主键快速定位完整行数据。
- 用非主键字段加速查询，同时通过主键回到聚簇索引取完整行。

## 核心机制
```text
1. 如果存在主键，主键索引就是聚簇索引。
2. 如果没有主键，选择第一个唯一索引作为聚簇索引。
3. 如果都没有，InnoDB 自动生成隐藏 rowid。
4. 二级索引查不到完整行时，需要拿主键再查聚簇索引。
```

## 重点部分详解
- 机制细节：聚簇索引叶子节点就是整行数据，而二级索引叶子节点只存索引列 + 主键。
- 常见故障：主键太大或频繁变动，会影响整表数据组织和二级索引成本。
- 典型代价：用二级索引查非覆盖列，通常会发生回表。
- 实战判断：主键设计不是“随便选一个字段”，它直接影响整张表的物理组织方式。

## 开发者视角
- 主键设计会影响整张表的数据组织。
- 使用二级索引查询非索引列时，可能发生 [[覆盖索引与回表]]。
- 主键建议短、稳定、递增，避免索引膨胀和频繁页分裂。

## 最小例子
```sql
select * from user where id = 10;
select * from user where name = 'Arm';
```

如果 `id` 是主键，第一条直接走聚簇索引；如果 `name` 是二级索引，第二条可能需要先找到主键，再回表取完整行。
