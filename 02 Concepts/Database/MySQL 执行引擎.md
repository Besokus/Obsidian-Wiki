# MySQL 执行引擎

## 一句话解释
执行引擎是 MySQL 用来实现存储引擎能力的操作层。对开发者来说，最重要的不是记住名字，而是理解不同引擎在事务、锁、索引和恢复上的差异。

## 它回答的核心问题
- 它试图解释：为什么同样是 MySQL，不同表或不同引擎的行为会不一样。
- 它试图约束：事务、锁、MVCC、崩溃恢复这些能力由谁提供。
- 它试图区分：MySQL 服务层和存储引擎层各自负责什么。

## 它解决什么问题
- 让数据库在不同场景下选择不同的存储与并发控制策略。
- 把业务读写请求转换成具体的存储操作。
- 为事务、恢复、锁控制等能力提供底层实现。

## 重点部分详解
- 机制细节：InnoDB 和 MyISAM 的差别，本质上是“能力边界”不同，不只是性能差异。
- 常见故障：把 MyISAM 当成支持事务的引擎使用，或者忽略表级锁带来的并发瓶颈。
- 典型代价：能力越强，引擎实现越复杂，维护成本和运行开销也通常越高。
- 实战判断：如果业务需要事务、崩溃恢复、行级并发控制，优先考虑 InnoDB。

## 核心机制
### InnoDB（默认）
- 支持事务，依赖 redo log 和 undo log。
- 支持行级锁和 MVCC。
- 使用聚簇索引组织数据。
- 支持崩溃恢复。

### MyISAM
- 不支持事务和外键。
- 主要提供表级锁。
- 读性能简单直接，但并发写能力弱。
- 崩溃恢复能力较弱。

## 对比
| 特性 | InnoDB | MyISAM |
|---|---|---|
| 事务 | 支持 | 不支持 |
| 锁粒度 | 行级 | 表级 |
| MVCC | 支持 | 不支持 |
| 外键 | 支持 | 不支持 |
| 崩溃恢复 | 支持 | 弱 |
| 适用 | 高并发事务 | 简单只读 |

## 易混概念
- [[MySQL 事务]]：事务是行为边界，执行引擎是实现边界。
- [[ACID]]：ACID 需要引擎能力支撑。
- [[InnoDB 锁]]：InnoDB 的并发控制能力主要靠锁实现。

## 我的理解
执行引擎不是一个纯“底层名词”，它直接决定数据库能否提供事务、并发和恢复能力。面试和排查时，先问引擎，再问功能，通常比直接背结论更稳。
