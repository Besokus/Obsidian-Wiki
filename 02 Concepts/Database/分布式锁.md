# 分布式锁

## 一句话解释
分布式锁用于协调多个服务实例对同一共享资源的并发访问，避免重复执行、重复扣减和竞态更新。

## 它回答的核心问题
- 它试图解释：为什么单机锁不够用。
- 它试图约束：多个进程、多个节点之间如何保证互斥。
- 它试图区分：锁服务、锁实现、业务幂等是不同层面的东西。

## 它解决什么问题
- 防止重复下单。
- 防止定时任务多实例重复执行。
- 防止库存、余额等共享资源被并发破坏。

## 重点部分详解
### Redis 分布式锁
- 常见写法：`SET lock:key uniqueValue NX EX 10`
- `NX` 表示仅当 key 不存在时设置成功。
- `EX` 表示自动过期，防止死锁。
- `uniqueValue` 用于标识锁持有者，避免误删。

### 正确释放
- 释放锁必须先校验 owner。
- 常用 Lua 脚本保证“校验 + 删除”原子性。

### 风险点
- 锁超时但业务没完成，会导致并发进入。
- 主从切换、网络抖动、GC pause 都可能影响锁可靠性。
- 获取锁失败后要有退避重试，而不是忙等。

## 如何避免死锁
- 锁必须设置 TTL。
- 解锁必须校验 owner。
- 多锁场景要固定加锁顺序。
- 需要可重入时，使用成熟组件，例如 Redisson。

## 它不适合解决的问题
- 它不是绝对可靠的全局一致性方案。
- 它不能替代幂等设计。
- 如果业务要求极高可靠性，可能更适合 ZooKeeper 或 etcd。

## 易混概念
- [[幂等性]]：即使锁失效，也要尽量保证重复执行不出错。
- [[Redis]]：Redis 可以实现锁，但不是锁语义本身。
- [[TCC]]：TCC 解决的是事务协同，不是互斥访问本身。

## 我的理解
分布式锁的价值是“减少并发冲突”，不是“把所有并发问题一次性解决”。真正稳定的方案通常是：锁 + 幂等 + 超时 + 补偿一起做。
