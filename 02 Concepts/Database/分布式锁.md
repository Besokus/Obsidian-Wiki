# 分布式锁

## 为什么需要分布式锁

多个服务实例同时操作同一资源时，需要互斥控制：

- 防止重复下单
- 防止定时任务多实例重复执行
- 防止库存并发扣减

## Redis 分布式锁

### 基本写法

```text
SET lock:key uniqueValue NX EX 10
```

- `NX`：key 不存在才设置成功
- `EX 10`：10 秒自动过期
- `uniqueValue`：锁持有者标识，防止误删

### 正确释放

用 Lua 脚本保证原子性：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
  return redis.call("del", KEYS[1])
else
  return 0
end
```

## 如何避免死锁

1. **锁必须设置 TTL**：防止持有锁的服务崩溃后锁永久存在
2. **解锁必须校验 owner**：防止误删别人的锁
3. **获取锁失败要有退避重试**：避免活锁
4. **多锁场景要固定获取顺序**：破坏循环等待条件
5. **可重入锁 + watchdog**：Redisson 等客户端提供自动续期

## 分布式锁的风险

Redis 分布式锁不是绝对可靠的：

- 锁过期业务未完成
- 主从切换导致锁丢失
- GC pause 导致锁误判

高可靠场景考虑 ZooKeeper 或 etcd。
