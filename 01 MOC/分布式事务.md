# 分布式事务

> 目标：能把一次跨服务业务（下单/支付/库存/优惠券）的失败路径讲清楚；能选对方案并做可恢复、可对账的工程落地。

## 快速入口
- 总览：[[分布式事务（入门整理）]]
- 协议/模式（原子笔记）
  - [[2PC]]（强原子提交，容易阻塞）
  - [[3PC]]（了解即可，工程少用）
  - [[TCC]]（资源预留：Try/Confirm/Cancel）
  - [[Saga]]（本地事务链 + 补偿）
  - [[Outbox Pattern]]（本地事务 + 可靠消息投递）
  - [[Transactional Message]]（事务消息：half message + 回查）
- 工程底座（原子笔记）
  - [[幂等性]]
  - [[补偿]]
  - [[事务状态机]]
  - [[对账]]

## 定位：你现在遇到的是什么问题？

| 现象/目标 | 优先看 | 关键词 |
|---|---|---|
| “跨库/跨资源必须强原子” | [[2PC]] | prepare/commit、阻塞、锁时间长 |
| “库存/优惠券/余额需要预留” | [[TCC]] | 幂等、空回滚、悬挂 |
| “长流程微服务，接受最终一致” | [[Saga]] | 编排/协同、补偿、弱隔离 |
| “本地事务后必须可靠通知下游” | [[Outbox Pattern]] / [[Transactional Message]] | 重复/延迟、消费者幂等 |
| “重试导致重复扣款/重复发货” | [[幂等性]] | 幂等 key、去重表、唯一约束 |
| “失败后怎么修回来、怎么复盘” | [[事务状态机]] / [[对账]] | 状态推进、超时、人工兜底 |

## 定位：典型电商链路如何拆

### 核心交易链路（强语义）
```text
创建订单
锁/扣库存（可预留）
冻结/核销优惠券（可预留）
支付（外部系统，不确定性强）
```
- 候选：[[TCC]]（库存/优惠券/余额）、[[Saga]]（长流程）、“本地事务 + [[事务状态机]]”

### 事件通知链路（可最终一致）
```text
发送订单创建消息
积分、搜索索引、推荐特征……
```
- 候选：[[Outbox Pattern]] / [[Transactional Message]] + [[幂等性]]

### 兜底链路（最后防线）
- [[对账]] + 补偿任务 + 人工处理入口（挂单/挂账）
